# Cambios 2026-05-17

## Rol Ansible `rdp` — fix DEFINITIVO: desplegar la config en vez de usar `grdctl`

### Hora: ~13:00

**Contexto**: el 2026-05-16 se intentó arreglar que el RDP quedara `disabled`
tras el primer arranque cambiando el rol a `parar daemon → grdctl set-*/enable
→ arrancar` + verificación. Tras regenerar la ISO y reinstalar, **siguió sin
funcionar**.

**Diagnóstico (equipo `10.0.72.51`, Ubuntu 26.04, gnome-remote-desktop
`50.0-0ubuntu2`)**:

1. Log `5-PrimerArranque.log` (12:24:25-26): el código nuevo corrió
   (`TASK [rdp : Parar gnome-remote-desktop antes de configurar]`), y la tarea
   `Verificar que el servidor RDP quedó habilitado` hizo
   `fatal: ... failed_when_result: true` con `Status: disabled`,
   `TLS certificate: (null)`, `TLS key: (null)`, **`Password: (hidden)`** →
   `PLAY RECAP ... failed=1` (consecuencia: `vscode` y roles posteriores **no
   se ejecutaron**).
2. Re-aplicando a mano con el daemon **vivo** (`set-* → enable →
   systemctl restart`): `Status: enabled`, persiste tras reboot.

**Causa raíz**: con el daemon `gnome-remote-desktop` **parado**,
`grdctl --system rdp set-credentials` SÍ persiste (almacén GKeyFile propio,
independiente del daemon) pero `set-tls-cert`/`set-tls-key`/`enable` **NO**
(necesitan el daemon vivo). El 1er fix paraba el daemon → rompía justo la parte
tls/enable. Confirmado por el patrón `Password: hidden` + `TLS: null` +
`disabled`.

**Ficheros de referencia capturados** (máquina donde funcionó y persistió tras
reboot):

- `/etc/gnome-remote-desktop/grd.conf` (owner `gnome-remote-desktop`, `0664`):
  ```ini
  [RDP]
  tls-cert=/etc/gnome-remote-desktop/tls.crt
  tls-key=/etc/gnome-remote-desktop/tls.key
  enabled=true
  ```
- `…/.local/share/gnome-remote-desktop/credentials.ini` (owner
  `gnome-remote-desktop`, `0600`) — **texto plano**, portable:
  ```ini
  [RDP]
  credentials={'username': <'ubuntu'>, 'password': <'ubuntu'>}
  ```

### Cambios aplicados (fix definitivo)

Estrategia: **no usar `grdctl` para configurar** (solo para leer/verificar);
escribir directamente los dos ficheros, que son deterministas, y reafirmarlos
en cada arranque antes del daemon (patrón `iac-gdm-noautologin.service`).

1. **`roles/rdp/templates/iac-rdp-config.sh.j2`** (nuevo): script que escribe
   `grd.conf` (`enabled=true` + rutas TLS, `0664`) y `credentials.ini`
   (GVariant con `rdp_username`/`rdp_password`, `0700` el dir, `0600` el
   fichero), todo owner `gnome-remote-desktop`.
2. **`roles/rdp/files/iac-rdp-config.service`** (nuevo): unidad `oneshot`,
   `DefaultDependencies=no`, `Before=gnome-remote-desktop.service`,
   `After=local-fs.target`, `WantedBy=gnome-remote-desktop.service`.
3. **`roles/rdp/tasks/main.yml`** reescrito: instala pkg → dir → cert →
   permisos cert/key → despliega script (`template`, `no_log`) → despliega
   unidad (`copy`) → `enable iac-rdp-config.service` (+`daemon_reload` si la
   unidad cambió) → lee `grdctl --system status` (idempotencia) → si no está
   ya `enabled`: parar daemon → ejecutar `iac-rdp-config.sh` → arrancar daemon
   → asegurar `started`+`enabled` → **verificación final** (`failed_when` si no
   quedó `enabled` con rutas TLS).
4. **`roles/rdp/defaults/main.yml`**: nueva var `rdp_creds_dir`
   (`/var/lib/gnome-remote-desktop/.local/share/gnome-remote-desktop`).
5. **`roles/rdp/CLAUDE.md`**: reescrito (nueva estrategia + historial de los
   dos bugs 16/17 para no repetirlos).
