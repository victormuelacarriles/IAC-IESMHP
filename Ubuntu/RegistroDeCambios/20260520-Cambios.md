## 1) Rol `nvidia`: sustituir el `reboot` comentado por una comprobación informativa

### Hora: ~10:30

**Contexto / petición**: el usuario comentó la tarea de `reboot` dentro del rol
[`Ubuntu/ansible/roles/nvidia/tasks/main.yml`](../ansible/roles/nvidia/tasks/main.yml)
porque al lanzarse desde `3-SetupPrimerInicio.sh` con `--connection=local` el
módulo `reboot` falla (intenta reconectarse via SSH a sí mismo y no hay sesión
remota). Pidió añadir **alguna comprobación en ese paso comentado** para
asegurar que las tareas siguientes (`nvidia-smi`, `ubuntu-drivers devices`) no
den problemas, **sin borrar el comentario**.

### Cambios aplicados

- **`Ubuntu/ansible/roles/nvidia/tasks/main.yml`**: tras el bloque comentado
  del `reboot`, se añaden dos tareas nuevas que actúan como sustituto:

  1. *«Comprobar si el módulo NVIDIA está cargado tras la instalación»* —
     `lsmod | awk '$1 ~ /^nvidia/ {print $1}'` con `failed_when:false` y
     `changed_when:false`. Registra en `nvidia_modules_loaded` los módulos
     NVIDIA activos en el kernel actual.
  2. *«Avisar si el módulo NVIDIA aún no está cargado»* — `debug` que se
     dispara cuando la salida del `lsmod` está vacía, dejando traza explícita
     en el log de que el driver se ha instalado pero **necesita el reinicio
     final** que hace `3-SetupPrimerInicio.sh` para activarse.

  El comentario del `reboot` original se mantiene **íntegro** (era un
  requisito explícito del usuario). Justo encima de las nuevas tareas se
  añade un bloque de comentario que explica:

  - Por qué el `reboot` está comentado (`--connection=local` + sin SSH propio).
  - Por qué no hace falta reiniciar aquí (`3-SetupPrimerInicio.sh` reinicia
    al final del primer arranque).
  - Qué hacen las tareas nuevas (asegurar que los pasos siguientes del rol
    no sorprenden con stdout vacío y dejar traza en el log).

### Notas / decisiones

- Las dos tareas posteriores del rol (`nvidia-smi --query-gpu...` y
  `ubuntu-drivers devices`) ya tenían `failed_when:false`, así que el rol no
  falla aunque el driver no esté cargado todavía. La aportación de las
  comprobaciones nuevas es **diagnóstica**: queda registrado en el log que
  el módulo aún no está cargado, lo que explica que `nvidia-smi` pueda
  devolver "no devices were found" durante este `ansible-playbook` (se
  resolverá en el reinicio final).

## 2) `3-SetupPrimerInicio.sh`: contador de intentos del bucle Ansible

### Hora: ~10:30

**Contexto / petición**: el usuario había añadido lógica para que, si
`ansible-playbook` devuelve un código de error, el script reinicie el equipo
**sin borrar el servicio**, de modo que en el siguiente arranque se vuelva a
ejecutar todo el proceso. Pidió añadir un **contador de reinicios**: máximo
3 intentos; al tercer fallo, detenerse y borrar el servicio de autoarranque
para evitar bucles infinitos.

### Cambios aplicados

- **`Ubuntu/ISO/26.04/3-SetupPrimerInicio.sh`** (`22.23-20260518` →
  `22.24-20260520`), bloque «autoconfiguración con Ansible»:

  - **Nuevo contador persistente** en `/var/lib/IAC-IESMHP/ansible-attempts`
    (variable `_ATTEMPT_FILE`). Se incrementa al inicio del bloque
    (`_ATTEMPT_NUM = leer-o-cero + 1`), se vuelca al fichero y se hace `sync`
    **antes** de lanzar `ansible-playbook` (si un reinicio interrumpiera el
    flush, el contador seguiría siendo correcto). `_MAX_ATTEMPTS=3`.
  - **Camino de fallo refactorizado**:
    - Si `_ATTEMPT_NUM < _MAX_ATTEMPTS`: avisa por `zenity`/`wall` que es el
      intento N/3 y reinicia (la rama original, intacta salvo el mensaje).
    - Si `_ATTEMPT_NUM >= _MAX_ATTEMPTS`: borra el contador, desactiva el
      servicio (`systemctl disable` + `rm` del `.service` + `mv "$0"
      "$0.borrado"` — el mismo triple mecanismo que la rama de éxito) y sale
      con `exit 1`. El equipo arranca normal en la siguiente sesión y queda
      accesible para diagnóstico manual.
  - **Camino de éxito**: además de la limpieza habitual del servicio, ahora
    también borra `_ATTEMPT_FILE` con `rm -f` (idempotente).

### Notas / decisiones

- `/var/lib/IAC-IESMHP/` se elige como ubicación del contador porque es la
  convención FHS para estado persistente de aplicaciones y sobrevive a
  reinicios (no se limpia como `/tmp` ni se rota como `/var/log`). El
  directorio se crea con `mkdir -p` por si es el primer uso.
- En el 3er fallo se usan `|| true` defensivos en `systemctl disable` y `mv`:
  si el servicio ya está deshabilitado o el script ya fue renombrado (caso
  raro pero posible si la rama de éxito se interrumpió a medias en un
  intento anterior), no queremos que el `exit 1` final se pierda.
- El comportamiento de la **rama de éxito** (intento 1, 2 o 3 que termina
  bien) no cambia: limpia servicio + ejecuta `4-Comprobaciones.sh` +
  reinicia.

## 3) `3-SetupPrimerInicio.sh`: garantizar que los errores de Ansible quedan en el log antes del reinicio

### Hora: ~10:30

**Contexto / petición**: el usuario detectó que cuando `ansible-playbook`
fallaba y el script reiniciaba el equipo, **el log `3-SetupPrimerInicio.sh.log`
se cortaba sin registrar el motivo**. El consumidor del log es una
process-substitution con `tee -a "$FLOG"` (definida al inicio del script): si
`systemctl reboot -i` se ejecuta antes de que esa subshell vacíe sus líneas
pendientes, el kernel apaga el equipo con las últimas líneas todavía en el
buffer del pipe → al rearrancar, el log no tiene la traza del fallo y no se
puede diagnosticar nada.

### Cambios aplicados

- **`Ubuntu/ISO/26.04/3-SetupPrimerInicio.sh`**, bloque del `ansible-playbook`:

  - Inmediatamente después de capturar `_RC_ANSIBLE=$?` se hace
    `echoamarillo "=== ansible-playbook devolvio rc=$_RC_ANSIBLE (intento N/M) ===" + sync`
    para grabar el resultado en disco **antes** de cualquier decisión.
  - En la rama de fallo (intento < máximo), antes del `sleep 10 &&
    systemctl reboot -i`:
    1. `sync` (vacía pages cache).
    2. `sleep 2` (el consumidor `tee` drena las líneas pendientes del pipe
       — la subshell del `exec > >( … )` necesita tiempo de CPU para leer
       lo que quedó en el buffer del pipe del kernel).
    3. `sync` otra vez (ya con todo escrito, fuerza a disco).
    4. El `sleep 10 && systemctl reboot -i` original (los 10 s de aviso al
       usuario por `zenity`/`wall` ya daban margen sobrado para el flush,
       pero los `sync`/`sleep` explícitos son la garantía dura).
  - En la rama de **3er fallo** (sin reinicio), también hay `sync` + `sleep 2`
    antes del `exit 1` por el mismo motivo (el EXIT trap del script hace su
    propio `sync`, pero el sleep adicional asegura que la última línea del
    `mostrar_mensaje` ha pasado por el `tee`).

### Notas / decisiones

- No se modifica el resto del pipeline de log (`exec > >(... tee -a "$FLOG"
  >&3)` + flusher periódico `while :; do sync; sleep 3; done`): es robusto
  contra cuelgues del kernel (su razón de ser, ver comentario al inicio del
  script). El problema era específico del camino que **provoca** el reinicio
  con error.
- `PYTHONUNBUFFERED=1` + `ANSIBLE_FORCE_COLOR=0` ya estaban exportados antes
  de `ansible-playbook` (no era el culpable: ansible sí imprimía el
  `fatal: ... FAILED!`, lo que se perdía era el `echorojo` y el rc del
  script que vienen **después** de la salida de ansible).
- La salida de `ansible-playbook` no se redirige explícitamente (hereda
  stdout/stderr del script, que ya pasan por el `tee`). No hace falta tocar
  esa parte.

## 4) `1-SetupLiveCD.sh`: FASE 1 ZFS — particionado y creación de zpool en el live CD (solo CEIABD)

### Hora: ~17:00

**Contexto / petición**: el usuario inicia la migración de `/home` (y en
CEIABD también `/datos`) de ext4 a ZFS con dedup + compresión inline. FASE 0
fue el descubrimiento del flujo (sin tocar nada); FASE 1 modifica únicamente
[`1-SetupLiveCD.sh`](../ISO/26.04/1-SetupLiveCD.sh) para crear el nuevo
particionado y los zpools antes del rsync. **Distancia se mantiene intacta**
en esta versión (decisión expresa del usuario al responder a las preguntas
sobre dedup/topología): solo CEIABD pasa a ZFS.

### Cambios aplicados

- **`Ubuntu/ISO/26.04/1-SetupLiveCD.sh`** (`22.2-20260428-Ubuntu` →
  `23.0-20260520-Ubuntu-zfs`):

  - **Detección de perfil explícita**: nueva variable `PERFIL` (`CEIABD` |
    `DISTANCIA`) fijada al detectar discos. Sustituye al patrón ad-hoc
    `[[ "$DISK_BIG" == *sd* ]]` que se repetía por todo el script. La
    cabecera printable y los logs declaran el perfil al inicio.
  - **Particionado bifurcado**:
    - **DISTANCIA**: bloque `parted` original sin cambios (EFI 512 MiB +
      swap 8 GiB + root resto ext4; disco grande NVMe 100 % ext4 → /home).
    - **CEIABD**: nuevo bloque `sgdisk` con tipos GPT correctos:
      | Part | Tamaño | Tipo | Destino |
      |------|--------|------|---------|
      | p1   | 1 GiB  | EF00 | EFI (FAT32) |
      | p2   | 16 GiB | 8200 | swap (partición, NO zvol) |
      | p3   | 100 GiB| 8300 | / (ext4) |
      | p4   | resto  | BF00 | zpool `rpool` → /home |
      Y el SATA grande: `p1` BF00 entera → zpool `tank` → /datos.
    - `partprobe` + `udevadm settle` sustituyen al `sleep 5` ciego.
    - `wipefs -a` adicional tras `sgdisk --zap-all` por si quedan firmas
      LVM/ZFS/mdadm residuales que `--zap-all` no toca.
  - **Asignación de particiones**: en CEIABD, además de los devnodes
    (`/dev/nvme0n1p4`, `/dev/sda1`), se resuelve `/dev/disk/by-id/...` con
    una función local `_resolver_byid` que prefiere `ata-*` / `nvme-MODELO-*`
    sobre `wwn-*` (que puede no existir en algunos firmwares NVMe). Si no
    encuentra ningún by-id, cae al devnode con aviso.
  - **Formateo**: en CEIABD se formatean solo EFI/swap/root; las BF00 las
    inicializa `zpool create`. Se generalizó el bucle `blkid` a `_UUID_CHECK`
    para no duplicar la lógica.
  - **Montaje**: en CEIABD solo se monta `/` y `/boot/efi` aquí; los
    datasets ZFS se montan dentro del bloque ZFS de más abajo.
  - **Nuevo bloque "ZFS (solo CEIABD)"**: lo lleva todo de punta a punta:
    1. `apt-get install -y zfsutils-linux` en el entorno live
       (`DEBIAN_FRONTEND=noninteractive`; el `update-initramfs` ya está
       enmascarado por `0b-Github.sh` así que la postinst no se cuelga).
    2. `modprobe zfs` y lectura de `zfs version`.
    3. Detección de **Fast Dedup** (OpenZFS ≥ 2.3): si está disponible se
       activa con `-o feature@fast_dedup=enabled` (~50 % menos RAM en la
       DDT respecto al dedup clásico); si no, dedup clásica con aviso.
    4. Limpieza de pools preexistentes (`zpool list` + `zpool import -d
       /dev/disk/by-id`) por si la VM se reinstala iterativamente.
    5. `zpool create rpool` con: `ashift=12`, `autotrim=on`,
       `cachefile=/etc/zfs/zpool.cache`, `acltype=posixacl`, `xattr=sa`,
       `atime=off`, `relatime=on`, `compression=zstd`, `dedup=on`,
       `recordsize=64K`, `-R /mnt`, sobre el `by-id` de la partición p4.
    6. `zfs create rpool/home` con `canmount=on mountpoint=/home`. **Nota
       FASE 1**: un único dataset (en vez de la estructura `rpool/home/<u>`
       del plan original) para que el `rsync` del paso siguiente vuelque
       /home (squashfs) al pool. FASE 2 (en `2-SetupSOdesdeLiveCD.sh`)
       refinará la estructura al crear el usuario `usuario`.
    7. `zpool create tank` análogo pero **sin dedup**, `recordsize=1M`,
       sobre el SDA entero.
    8. `zfs create tank/datos` con `mountpoint=/datos`, `setuid=off`,
       `devices=off`; `chmod 1777` para hacerlo área de datos compartida.
    9. Diagnóstico final: `zpool status` + `zfs list`.
  - **Copia de `zpool.cache`**: tras el `rsync` (importante: si se hace
    antes, el rsync del squashfs lo sobreescribiría), se copia
    `/etc/zfs/zpool.cache` a `/mnt/etc/zfs/zpool.cache` para que
    `zfs-mount.service` del sistema instalado importe los pools sin
    escanear `/dev/disk` al arrancar.
  - **`.iac-partitions.env` ampliado**: ahora incluye `PERFIL=` y, solo en
    CEIABD, `ZFS_POOL_HOME=rpool`, `ZFS_HOME_DATASET=rpool/home`,
    `ZFS_HOME_PARTID=<by-id>`, `ZFS_POOL_DATA=tank`,
    `ZFS_DATA_DATASET=tank/datos`, `ZFS_DATA_PARTID=<by-id>`. `PART_DATA`
    queda vacío en CEIABD como marcador de escenario ZFS para
    `2-SetupSOdesdeLiveCD.sh`. Distancia escribe exactamente las mismas 4
    variables que antes (`PART_EFI`, `PART_SWAP`, `PART_ROOT`,
    `PART_DATA`).
  - **Export de pools antes del reboot**: si `2-SetupSOdesdeLiveCD.sh`
    termina con "Correcto", antes del `reboot` se desmontan los bind-mounts
    virtuales del chroot (`/mnt/dev/pts`, `/mnt/run`, `/mnt/sys`,
    `/mnt/proc`, `/mnt/dev`), se hace `zpool sync` y `zpool export rpool`
    + `zpool export tank`. Sin esto el sistema instalado vería los pools
    marcados "in use" por el hostid del live y `zfs-import-cache` tendría
    que importar con `-f`. En la rama de fallo NO se exporta: queremos que
    los pools sigan accesibles desde `/mnt` para diagnóstico manual.

### Decisiones y notas

- **Distancia intacta** por petición expresa del usuario al cerrar las
  preguntas de FASE 0. Esta versión solo lleva ZFS a CEIABD; la rama
  Distancia ejecuta byte a byte el código original (`parted`, ext4, mismas
  variables en `.iac-partitions.env`).
- **swap en partición, no en zvol**: regla del plan original, evita el
  deadlock ARC↔swap documentado por OpenZFS.
- **`-R /mnt` (altroot)** en ambos `zpool create`: los datasets se montan
  bajo `/mnt/*` durante el live; tras `export` + arranque del sistema
  instalado, los mountpoints reales son `/home` y `/datos`.
- **`cachefile=/etc/zfs/zpool.cache`** en lugar de `cachefile=none`: con
  cache el sistema instalado importa los pools de forma determinista y más
  rápida que escaneando `/dev/disk` al arranque.
- **`zfs-initramfs` y `zfs-dkms`**: NO se instalan en el live (no hace
  falta para el flujo de FASE 1: el módulo `zfs` ya viene en el kernel del
  live). Su instalación corresponde a FASE 2 dentro del chroot, junto con
  el ajuste de `/etc/fstab` y la creación de datasets por usuario.
- **Compatibilidad con `2-SetupSOdesdeLiveCD.sh` actual**: ese script lee
  `PART_DATA` y monta `/home` o `/datos` ext4 según el tipo de disco. En
  CEIABD ahora `PART_DATA=` (vacío), por lo que `2-...` tomará la rama de
  fallback con `lsblk`, que NO encontrará `/mnt/home` ni `/mnt/datos`
  porque son datasets ZFS (no aparecen en `lsblk -rno NAME,MOUNTPOINT` con
  formato esperado). **Esto romperá la generación de `/etc/fstab` en CEIABD
  en esta versión** — es deliberado, FASE 2 reescribirá ese bloque para
  detectar `ZFS_*` y emitir un fstab sin entrada `/home` ni `/datos` (los
  monta `zfs-mount.service`). Hasta que llegue FASE 2, esta versión del
  script 1 **no es aún apta para CEIABD en producción**: solo Distancia
  arranca a sistema completo.

### Pendiente para FASE 2 (próximo turno)

- Adaptar `2-SetupSOdesdeLiveCD.sh` para detectar `ZFS_*` en
  `.iac-partitions.env` y:
  1. Instalar `zfsutils-linux` + `zfs-initramfs` + `zfs-dkms` en el chroot
     con DKMS diferido al primer arranque.
  2. Emitir `/etc/fstab` SIN entradas `/home` ni `/datos` cuando hay ZFS.
  3. Crear `rpool/home/usuario` antes del `useradd` y mover contenidos.
  4. Asegurar que `update-initramfs -c -k all` incluye el módulo ZFS.
- `4-Comprobaciones.sh`: nueva sección 9 "ZFS" con `zpool status`,
  `zfs list`, dedup ratio, `lsinitramfs | grep zfs`.

## 6) `2-SetupSOdesdeLiveCD.sh`: FASE 2 ZFS — configuración del SO y helper de alta de usuarios

### Hora: ~17:45

**Contexto / petición**: con `1-SetupLiveCD.sh` ya preparado para crear los
zpools (FASE 1, sección 4 de este registro), toca adaptar el script de chroot
para que el sistema instalado quede operativo en ZFS: paquetes, servicios,
fstab sin entradas `/home`/`/datos`, dataset por usuario y helper para dar
de alta alumnos.

### Cambios aplicados

- **`Ubuntu/ISO/26.04/2-SetupSOdesdeLiveCD.sh`** (`22.23-20260519` →
  `23.0-20260520-zfs`):

  - **Preámbulo**: carga `/tmp/.iac-partitions.env` al inicio (antes del
    primer `paso`), de forma que la variable `PERFIL` (CEIABD | DISTANCIA)
    queda disponible para todos los bloques. Si el fichero no existe (caso
    de re-ejecución manual del script desde un sistema ya instalado), se
    asume `PERFIL=DISTANCIA` por compatibilidad.

  - **fstab bifurcado por perfil**:
    - **DISTANCIA**: igual que antes (4 líneas: `/`, `/boot/efi`, `/home`
      o `/datos` ext4, `swap`).
    - **CEIABD**: solo 3 líneas (`/`, `/boot/efi` y `swap`). `/home` lo
      monta `rpool/home/<usuario>` vía `zfs-mount.service`; `/datos` lo
      monta `tank/datos`. Se añaden comentarios al inicio del fstab
      explicando qué dataset cubre cada punto de montaje. La partición
      raíz lleva `noatime` (con ZFS aparte, no se necesita atime en /).

  - **Nuevo bloque "Instalar ZFS en el sistema instalado y habilitar
    servicios"** (solo CEIABD), insertado tras "Verificar conectividad":
    1. `apt-get install linux-headers-generic` (requerido por `zfs-dkms`).
    2. `apt-get install zfsutils-linux zfs-zed zfs-dkms zfs-initramfs`
       con `DEBIAN_FRONTEND=noninteractive` y `--force-confold`.
    3. `systemctl enable zfs.target zfs-import-cache zfs-mount zfs-zed`.
    4. `systemctl disable zfs-import-scan` (redundante con cachefile y
       penaliza el tiempo de boot).
    5. `zpool set cachefile=/etc/zfs/zpool.cache rpool` y `tank` para
       refrescar el cachefile desde los binarios del sistema instalado.
    6. Diagnóstico final: `zpool status` + `zfs list`.

  - **Reestructuración de `rpool/home`** en el bloque "Usuarios":
    - En FASE 1 el script `1-SetupLiveCD.sh` creó un único dataset
      `rpool/home` con `canmount=on mountpoint=/home` (para que el rsync
      vaciara `/home/ubuntu` al pool). Ahora, antes del `useradd`:
      - Si el dataset existe, `zfs umount rpool/home` + `zfs destroy -r
        rpool/home` (eliminamos el contenido del rsync; el usuario
        `ubuntu` del live lo borra completo el paso siguiente).
      - `zfs create -o canmount=off -o mountpoint=/home rpool/home`
        → contenedor para datasets por usuario.
      - `zfs create -o canmount=on rpool/home/usuario` con `quota=40G`.
        `mountpoint` heredado del padre → `/home/usuario`.

  - **`useradd` adaptado**: en CEIABD se llama con `-d /home/usuario`
    (el directorio existe como dataset montado vacío) y sin `-m` ambiguo;
    se copia `/etc/skel` a mano si el dataset quedó vacío tras el
    `useradd`, y se hace `chown -R usuario:usuario /home/usuario`. En
    DISTANCIA se conserva el `useradd -m -s /bin/bash -p '*' usuario`
    original.

  - **Snapshot `rpool/home/usuario@inicial`**: tras configurar contraseña
    y copiar `authorized_keys`. Es ancla de rollback "estado justo tras la
    instalación" y muestra del flujo que aplicará el helper.

  - **Nuevo paso "Helper `/usr/local/sbin/nuevo-alumno.sh`"** (solo
    CEIABD): generado inline con heredoc. Acepta `<usuario> [cuota=40G]`
    y hace:
    1. Validaciones (root, pool existe, usuario no existe, dataset no
       existe, nombre conforme).
    2. Detecta grupos opcionales presentes (`vboxusers`, `libvirt`,
       `docker`) y los añade a la lista; `sudo` siempre.
    3. `zfs create rpool/home/<u>` + `zfs set quota=<cuota>`.
    4. `useradd -k /etc/skel -d /home/<u> -s /bin/bash -G <grupos>`.
    5. Copia `/etc/skel` si el dataset quedó vacío + `chown` recursivo.
    6. `zfs snapshot rpool/home/<u>@inicial`.
    7. `passwd <u>` interactivo.

  - **NO se ha tocado**: GDM, GRUB, machine-id, plymouth, casper hooks,
    update-initramfs ni la unidad `3-SetupPrimerInicio.service`. La
    función `update-initramfs -c -k all` del bloque "casper" ya recoge
    `zfs-initramfs` automáticamente al estar instalado antes en este
    mismo script.

### Decisiones y notas

- **Estructura padre-hijos `rpool/home`**:
  - `rpool/home` contenedor → `canmount=off`, `mountpoint=/home`. NO se
    monta, así que `/home` es el dir real del root ext4 (necesario para
    que existan los puntos de montaje de los hijos).
  - `rpool/home/<usuario>` por usuario → `canmount=on`, mountpoint
    heredado `/home/<usuario>`. Permite cuota individual y snapshots
    `@inicial`/`@diario` por usuario.
- **`useradd -d <home>` sin `-m`** en CEIABD: con `-m` y home preexistente
  (caso de dataset ya montado), `useradd` no siempre copia `skel`. La

## 8) `2-SetupSOdesdeLiveCD.sh`: deshabilitar `os-prober` para evitar cuelgue en chroot (CEIABD)

### Hora: ~22:30

**Contexto / síntoma**: en la primera ejecución de la versión ZFS
(`23.0-20260520-zfs`) sobre VMware con perfil CEIABD, el script se queda
bloqueado **indefinidamente** dentro del PASO 9 ("Instalar ZFS en el sistema
instalado"). El log `1-SetupLiveCD.sh.log` se corta justo en:

```
Procesando disparadores para linux-image-7.0.0-15-generic (7.0.0-15.15) ...
/etc/kernel/postinst.d/initramfs-tools:
update-initramfs: Generating /boot/initrd.img-7.0.0-15-generic
...
/etc/kernel/postinst.d/kdump-tools:
W: kdump-tools: Executing in a chroot, skipping initramfs generation.
```

…y no avanza más (no llega a `info "Instalando paquetes ZFS..."`).

### Causa raíz

1. `apt-get install linux-headers-generic` arrastra los meta-paquetes
   `linux-generic-hwe-26.04` (upgrade) → instala **kernel nuevo**
   `linux-image-7.0.0-15-generic` y el módulo precompilado
   `linux-main-modules-zfs-7.0.0-15-generic` (módulo ZFS firmado por
   Canonical — DKMS realmente innecesario, optimización aparte).
2. El postinst del kernel ejecuta `/etc/kernel/postinst.d/*` en orden:
   `initramfs-tools` (OK, visto) → `kdump-tools` (OK, último visto) →
   `unattended-upgrades` → `update-notifier` → `zz-flash-kernel` →
   **`zz-update-grub`** ← aquí cuelga.
3. `update-grub` invoca **`os-prober`** por defecto. En chroot, `/dev` está
   bind-mounted desde el live → os-prober ve todo: ~16 loops snap, `sr0`,
   `nvme0n1p1..p4`, `sda1`. Y además **los zpool `rpool` y `tank` están
   importados** con altroot=/mnt (los creó FASE 1).
4. os-prober intenta montar cada bloque en RO para fingerprintearlo. Con
   ZFS importado por encima de las particiones BF00 + loops snap activos,
   se bloquea (no falla, no avanza).

### Fix aplicado

Forzar `GRUB_DISABLE_OS_PROBER=true` en `/etc/default/grub` **antes** del
primer `apt-get install` del bloque ZFS, justo después del comentario que
abre el bloque y antes del `apt-get update`. Así:

- El `update-grub` implícito disparado por el postinst del kernel nuevo
  no lanza os-prober → no se cuelga.
- El `update-grub` explícito del paso "GRUB" más abajo
  ([2-SetupSOdesdeLiveCD.sh:989](../ISO/26.04/2-SetupSOdesdeLiveCD.sh#L989))
  hereda el mismo ajuste (no había riesgo análogo allí porque entonces
  ZFS ya está instalado en el sistema, pero conceptualmente igual de
  innecesario).
- En el sistema instalado, os-prober queda permanentemente deshabilitado.
  Es lo correcto: estos equipos solo arrancan Ubuntu, no hay dual-boot
  que detectar — os-prober solo añadiría ruido en cada `update-grub`.

```bash
if grep -q '^GRUB_DISABLE_OS_PROBER=' /etc/default/grub; then
    sed -i 's/^GRUB_DISABLE_OS_PROBER=.*/GRUB_DISABLE_OS_PROBER=true/' /etc/default/grub
else
    echo 'GRUB_DISABLE_OS_PROBER=true' >> /etc/default/grub
fi
```

### Por qué solo en CEIABD

El fix se aplica DENTRO del `if [ "$PERFIL" = "CEIABD" ]; then` del PASO 9
porque ese bloque es el que dispara la cadena de cuelgue (es el único que
hace `apt-get install` antes del PASO "casper"/initramfs/grub finales).
DISTANCIA no instala kernel adicional en chroot → no llega a este caso.
Si en el futuro DISTANCIA hiciera un install que upgradeara el kernel,
habría que mover esta línea fuera del `if`.

### Verificación previa al fix

Revisada toda la carpeta `Ubuntu/RegistroDeCambios/` (11 ficheros del
2026-04-30 al 2026-05-20): **ningún cambio anterior** ha tocado
`GRUB_DISABLE_OS_PROBER` ni `os-prober`. Las modificaciones GRUB de
`20260504-Cambios.md` (`GRUB_TIMEOUT=5`, `GRUB_TIMEOUT_STYLE=menu`,
`GRUB_DISABLE_SUBMENU=y`, `11_iac_texto`) actúan sobre claves distintas
y no se pisan con esta línea — se ejecutan en el paso "GRUB" posterior
con `sed` específicos.

### Ficheros modificados

- **`Ubuntu/ISO/26.04/2-SetupSOdesdeLiveCD.sh`** (`23.0-20260520-zfs` →
  `23.1-20260520-zfs`): bloque nuevo justo antes de `apt-get update -y`
  del PASO 9, sin alterar el resto del flujo.
- **`Ubuntu/RegistroDeCambios/20260520-Cambios.md`** (esta sección).

## 9) `2-SetupSOdesdeLiveCD.sh`: deshabilitar kernel-postinst hooks durante install ZFS

### Hora: ~23:15

**Contexto**: el fix anterior (sección 8: `GRUB_DISABLE_OS_PROBER=true`) **no
resolvió el cuelgue**. La segunda ejecución (vs `23.1-20260520-zfs`) muestra
que la línea se aplica correctamente (`[INF] 17:49:58 GRUB_DISABLE_OS_PROBER=
true en /etc/default/grub`) pero el install se vuelve a bloquear **exactamente
en el mismo punto** del log: tras `W: kdump-tools: Executing in a chroot,
skipping initramfs generation` no hay más salida.

### Re-análisis

Como `update-grub`/`os-prober` ya no es candidato, el cuelgue debe estar en
algún hook de `/etc/kernel/postinst.d/` **posterior** a `kdump-tools`. En
Ubuntu 26.04, los hooks típicos en ese directorio (en orden alfabético, que
es el orden de ejecución por `run-parts`):

```
apt-auto-removal
initramfs-tools       ← OK (visto en log)
kdump-tools           ← OK (último visto en log)
unattended-upgrades   ← sospechoso #1: intenta hablar con apt daemon / D-Bus
update-notifier       ← sospechoso #2: ídem
zz-flash-kernel       ← sospechoso #3: para ARM/embedded; en chroot x86 raras
                        veces se cuelga buscando DTBs
zz-update-grub        ← improbable ya (OS_PROBER off) pero por seguridad
```

Sin más timestamps en la salida apt, no se puede saber cuál de los 4 es el
culpable exacto. Todos son seguros de saltar en este contexto:

- `update-grub` se ejecuta explícitamente más abajo en el paso "GRUB"
  ([2-SetupSOdesdeLiveCD.sh:989](../ISO/26.04/2-SetupSOdesdeLiveCD.sh#L989)),
  así que saltarlo aquí no causa daño.
- `unattended-upgrades` y `update-notifier` solo afectan notificaciones
  cosméticas para el usuario y al estado del auto-upgrade — no rompen
  arranque ni instalación.
- `flash-kernel` es no-op en x86_64 PC (no hay DTB que escribir).

### Fix aplicado

`chmod -x` los 4 hooks ANTES del primer `apt-get install` del bloque ZFS;
`chmod +x` para restaurarlos al final del bloque, antes de salir del
`if [ "$PERFIL" = "CEIABD" ]; then`. Así:

- Durante los 2 `apt-get install` (linux-headers-generic + paquetes ZFS),
  `run-parts /etc/kernel/postinst.d/` salta los 4 hooks porque no son
  ejecutables → no se ejecuta nada que pueda colgarse.
- Tras los installs, restauramos `+x` → el sistema instalado tendrá los
  hooks operativos cuando ejecute `apt upgrade` en futuros arranques.
- Se llevan a `_KHOOKS_DISABLED[]` solo los hooks que SÍ estaban `-x`
  inicialmente (defensivo: si algún hook no existe en este sistema,
  no se intenta restaurar uno inexistente).

```bash
_KHOOKS_DIR=/etc/kernel/postinst.d
_KHOOKS_DISABLED=()
for _h in unattended-upgrades update-notifier zz-flash-kernel zz-update-grub; do
    if [ -x "$_KHOOKS_DIR/$_h" ]; then
        chmod -x "$_KHOOKS_DIR/$_h"
        _KHOOKS_DISABLED+=("$_h")
    fi
done
# ... 2× apt-get install ...
for _h in "${_KHOOKS_DISABLED[@]}"; do
    [ -e "$_KHOOKS_DIR/$_h" ] && chmod +x "$_KHOOKS_DIR/$_h"
done
```

### Robustez ante fallos

- El `apt-get install linux-headers-generic` recibe ahora `|| err "..."`
  para que `set -e` no aborte sin restaurar (antes solo lo tenía el
  segundo install ZFS).
- Si los apt fallan, `err` solo loguea — el bucle de restauración corre
  siempre. Los hooks quedan operativos en el sistema instalado sea cual
  sea el resultado.

### Por qué `chmod -x` y no `dpkg-divert`

`dpkg-divert` es más "oficial" pero requiere recordar el rename exacto y
deshacerlo con `--remove`. Si algo va mal a medias, queda un fichero
huérfano `.distrib` en el sistema instalado. `chmod -x` es trivialmente
reversible (solo el bit ejecutable) y `run-parts` ignora ficheros
no-ejecutables por diseño — no hay efectos secundarios.
