## 09:09 — Centralización de variables "mágicas" en `comun.sh` (única fuente de verdad)

**Motivo**: las mismas rutas y constantes (`REPO="IAC-IESMHP"`, la URL de
GitHub `https://github.com/victormuelacarriles/IAC-IESMHP.git`, `/opt/IAC-IESMHP`,
`/var/log/IAC-IESMHP/Ubuntu`, `DISTRO="Ubuntu"`, `versionDISTRO`, rutas de los
sub-scripts, URL raw de `macs.csv`, redes de aula `10.0.72`/`10.0.32`...) estaban
**duplicadas a mano en casi todos los scripts** de la cadena. Cambiar el repo de
sitio o renombrar una ruta obligaba a editar 6-7 ficheros y era fácil dejar uno
descuadrado (de hecho así nació el bug Mint del 2026-06-11).

**Solución**: nuevo fichero **`Ubuntu/ISO/26.04/comun.sh`** que define **una sola
vez** todas esas variables. El resto de scripts hacen `source` de él en lugar de
redefinirlas.

### Fichero nuevo

- **`Ubuntu/ISO/26.04/comun.sh`** — define:
  - Identidad del repo: `GITHUB_USER`, `REPO`, `RAMA`, `GITREPO`, `GITRAW`.
  - Distro: `DISTRO`, `versionDISTRO` (de `/etc/os-release`, fallback `26.04`).
  - Rutas: `RAIZSCRIPTS` (`/opt/IAC-IESMHP`), `RAIZDISTRO`, `RAIZUTILES`,
    `RAIZANSIBLE`, `RAIZLOG` (`/var/log/IAC-IESMHP/Ubuntu`).
  - Rutas absolutas de los sub-scripts: `SCRIPT_LIVECD`, `SCRIPT_CHROOT`,
    `SCRIPT_PRIMERINICIO`, `SCRIPT_COMPROBACIONES`, `SCRIPT_NOMBREIP`,
    `SCRIPT_AUTOANSIBLE`.
  - Ficheros de datos: `FICHERO_MACS`, `URL_MACS`, `FICHERO_AUTORIZADOS`.
  - Redes/proxy por aula: `RED_IABD`/`RED_SMRD`, `PROXY_IABD`/`PROXY_SMRD`.
  - Lleva una **guarda de doble carga** (`IAC_COMUN_CARGADO`) y NO se ejecuta:
    solo se carga con `source`. No fija `set -e` (no toca el shell padre).

### Patrón de carga

Cada script localiza `comun.sh` **relativo a su propia ruta** (vía
`${BASH_SOURCE[0]}`), sin codificar `/opt/IAC-IESMHP` a fuego:

```bash
# scripts en ISO/26.04/
_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"; source "$_DIR/comun.sh"
# scripts en ISO/26.04/utiles/
_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"; source "$_DIR/comun.sh"
```

Todos los scripts de la cadena se invocan con ruta absoluta (0b → `bash`,
chroot, systemd `ExecStart`, `/bin/bash`), por lo que `dirname "${BASH_SOURCE[0]}"`
siempre resuelve bien. Verificado.

### Scripts modificados

- **`0b-Github.sh`** — es la **ÚNICA excepción**: corre en el Live CD *antes* de
  clonar el repo, así que no puede cargar `comun.sh` todavía. Conserva un
  pequeño **bloque de arranque** con solo `GITHUB_USER` + `REPO` (claramente
  marcado, "DEBE COINCIDIR con comun.sh"); deriva `GITREPO`/`DESTDIR`/`RAIZLOG`
  y, **tras clonar**, carga `comun.sh` para obtener `SCRIPT_INSTALL`
  (= `SCRIPT_LIVECD`), eliminando el `26.04` codificado a fuego. Tiene fallback
  por si el repo clonado fuese antiguo y no tuviera `comun.sh`.
- **`1-SetupLiveCD.sh`** — fuera `REPO`/`GITREPO`/`DISTRO`/`RAIZSCRIPTS`/
  `RAIZLOG`/`versionDISTRO`; ahora `source comun.sh`. Se conserva
  `RAIZSCRIPTSLIVE="$RAIZSCRIPTS"` (alias) y `SCRIPT2=$(basename "$SCRIPT_CHROOT")`.
- **`2-SetupSOdesdeLiveCD.sh`** — fuera la cabecera de variables; `source comun.sh`
  (funciona dentro del chroot, ruta absoluta). `SCRIPT3=$(basename "$SCRIPT_PRIMERINICIO")`.
- **`3-SetupPrimerInicio.sh`** — fuera `REPO`/`DISTRO`/`RAIZSCRIPTS`/`RAIZLOG`/
  `RAIZDISTRO`/`RAIZANSIBLE`/`versionDISTRO`; `source comun.sh`.
  `SCRIPT4nombreip`/`SCRIPT5ansible` ahora apuntan a `$SCRIPT_NOMBREIP` /
  `$SCRIPT_AUTOANSIBLE`.
- **`4-Comprobaciones.sh`** — fuera `REPO`/`DISTRO`/`RAIZLOG`; `source comun.sh`.
- **`utiles/NombreIP.sh`** — fuera `REPO`/`DISTRO`/`RAIZSCRIPTS`/`RAIZLOGS` y la
  `URL_MACS` hardcodeada; `source ../comun.sh`. `RAIZLOGS="$RAIZLOG"` (alias).
  La comparación de aula usa `$RED_IABD`/`$RED_SMRD` en vez de `10.0.72`/`10.0.32`.

`utiles/Auto-Ansible.sh` y `0a-CreaISO.sh` **no se tocan**: no contienen
variables repetidas del proyecto (el primero solo gestiona claves SSH; el
segundo es el generador de ISO y sus rutas son de construcción, no de `/opt`).

### Verificación realizada

- `bash -n` (sintaxis) OK en los 8 scripts.
- `source comun.sh` bajo `set -euo pipefail` carga sin abortar y expande
  correctamente todas las variables.
- Patrón de `source` relativo probado desde `ISO/26.04/` y desde `utiles/`, e
  invocando por ruta absoluta (como systemd/chroot).

### ⚠️ Importante para el despliegue

`comun.sh` es un fichero **nuevo**: hay que **commit + push a `main`** antes de
generar/arrancar una ISO, porque `0b-Github.sh` clona el repo desde GitHub y
buscará `comun.sh` en el clon. Si por lo que sea no estuviera, `0b-Github.sh`
tiene fallback a la ruta directa de `1-SetupLiveCD.sh` (no rompe), pero el
resto de scripts (1-4, NombreIP) **sí lo necesitan** para arrancar.

### Cómo cambiar una ruta a partir de ahora

Editar **solo** `Ubuntu/ISO/26.04/comun.sh`. Si se cambia la URL del repo o el
nombre `REPO`, recordar actualizar **también** el bloque de arranque de
`0b-Github.sh` (las dos líneas `GITHUB_USER`/`REPO`), que es la única
duplicación inevitable.

## 09:52 — Simplificación ZFS: `/home` pasa a ser un dataset ÚNICO (sin datasets por usuario, sin cuotas, sin helper)

**Motivo**: petición del usuario — simplificar la gestión ZFS. La estructura
contenedor (`rpool/home canmount=off`) + un dataset por usuario
(`rpool/home/usuario`, `rpool/home/<alumno>` vía `nuevo-alumno.sh`) añadía
complejidad (re-import sin altroot en chroot, redes de seguridad por dataset,
helper de alta propio) sin aportar nada una vez desactivadas las cuotas
(2026-06-09). La compresión y la dedup son propiedades del pool y aplican
igual con un solo dataset.

**Estructura nueva (CEIABD)**:

```
rpool/home   (canmount=on, mountpoint=/home — dataset único, SIN cuotas)
tank/datos   (sin cambios)
```

Todos los usuarios crean su perfil como **directorio normal** dentro de
`/home`. Alta de usuarios: `sudo adduser <usuario>` (flujo estándar de Linux).

### `2-SetupSOdesdeLiveCD.sh` (`23.6` → `23.7-20260612-zfs-homeunico`)

- **Eliminado** el bloque de reestructuración del paso "Usuarios": ya no se
  destruye `rpool/home`, ya no se recrea como contenedor `canmount=off`, ya no
  se crea `rpool/home/usuario`.
- **Eliminado** el re-import de `rpool` sin altroot dentro del chroot: existía
  solo porque se creaban datasets nuevos en el chroot (bug 2026-05-20 de la
  altroot); al no crear datasets, no aplica. El dataset `rpool/home` llega
  montado en `/home` desde el live (vía altroot `/mnt`).
- **Conservada** una red de seguridad mínima: si `PERFIL=CEIABD` y `rpool`
  existe, `mountpoint -q /home` o aborta (evita escribir al ext4 subyacente
  oculto bajo el dataset).
- `useradd` unificado: la rama CEIABD usa ahora el mismo
  `useradd -m -s /bin/bash -p '*' usuario` que DISTANCIA (antes: `useradd`
  sin `-m` + copia manual de skel + chown por ser el home un mountpoint ZFS).
- **Snapshot** `rpool/home/usuario@inicial` → sustituido por un snapshot
  global **`rpool/home@inicial`**, tomado DESPUÉS de eliminar el usuario
  `ubuntu` del Live CD (antes el snapshot se tomaba antes de ese paso; con el
  dataset único habría arrastrado `/home/ubuntu`). El rollback ahora afecta a
  TODO `/home` — documentado con aviso.
- **Eliminado** el helper `/usr/local/sbin/nuevo-alumno.sh` (≈100 líneas de
  heredoc): sin datasets por usuario no tiene función. Queda un comentario en
  su lugar. En equipos ya instalados el helper antiguo puede seguir presente —
  no usarlo.
- Con esto desaparecen también las dos líneas comentadas `zfs set quota=…`
  (estaban dentro de los bloques eliminados). Si algún día se quieren cuotas
  con dataset único, usar cuotas de usuario POSIX de ZFS:
  `zfs set userquota@<u>=200G rpool/home`.

### `1-SetupLiveCD.sh` (`23.0` → `23.1-20260612-Ubuntu-zfs`)

- Solo comentarios: el dataset `rpool/home` que crea (canmount=on,
  mountpoint=/home) deja de ser "FASE 1 provisional" y pasa a ser el dataset

## 16:05 — Nuevo rol Ansible `preparaAD` (prerequisitos de unión a dominio Active Directory)

**Motivo**: petición del usuario — dejar el equipo preconfigurado para unirse
a un dominio AD (Ubuntu Desktop 26.04). Cumple el TODO `predominio` que estaba
anotado en `roles.yaml`.

### Rol nuevo: `Ubuntu/ansible/roles/preparaAD/`

Prerequisitos implantados (doc oficial Ubuntu, SSSD with Active Directory):

- **Paquetes**: `realmd`, `sssd-ad`, `sssd-tools`, `libnss-sss`, `libpam-sss`,
  `adcli`, `krb5-user`, `packagekit`, `bind9-dnsutils`.
- **`pam_mkhomedir`** activado con `pam-auth-update --enable mkhomedir`
  (home automático en el primer login de un usuario del dominio; solo corre
  si no estaba ya en `common-session`).
- **Reloj**: Kerberos exige < 5 min de desfase con el DC. Variable opcional
  `preparaad_ntp` (drop-in de timesyncd + restart); siempre se informa de
  `NTPSynchronized` en el resumen.
- **Hostname**: aviso si > 15 caracteres (límite NetBIOS de AD).
- **`/etc/krb5.conf`** por plantilla (solo si se conoce el dominio): realm
  por defecto, KDC por DNS (SRV), `rdns=false`.
- **Comprobación de unión**: `realm list --name-only` (¿ya unido?) +
  `realm discover` informativo (¿el DNS del aula resuelve el dominio?).
- **Unión opcional** (apagada por defecto, `preparaad_unir=false`):
  `realm join --unattended` con contraseña por stdin (`no_log`),
  `--computer-ou` opcional y post-condición ruidosa (falla si `realm list`
  sigue vacío).
- **Snippet `/etc/sssd/conf.d/10-iac-ad.conf`** SOLO con el equipo ya unido:
  `use_fully_qualified_names=False`, `fallback_homedir=/home/%u`,
  `cache_credentials=True`, `ad_gpo_access_control=permissive` (el default
  `enforcing` de SSSD bloquea logins si las GPO no contemplan Linux).
  Pre-unión `conf.d/` queda vacío a propósito: el `sssd.service` de Ubuntu
  lleva `ConditionPathExists=/etc/sssd/sssd.conf` +
  `ConditionDirectoryNotEmpty=/etc/sssd/conf.d/` y así se salta limpiamente
  (sin unidad fallida que dispare la sección 8 de `4-Comprobaciones.sh`).

Respeta los cuelgues documentados 2026-05-17: el módulo `systemd` solo se usa
con `state: restarted` (nunca `enabled:`/`daemon_reload:`).

### Cómo unir (documentado en `roles/preparaAD/CLAUDE.md`)

Recomendado: cuenta AD **delegada** solo para alta de equipos en una OU
(`svc-union-linux`) + credenciales en **ansible-vault**
(`group_vars/all/vault.yml` cifrado, committeable) + pase de aula manual:
`ansible-playbook -i equiposIABD.ini roles.yaml --tags preparaad
-e preparaad_unir=true --ask-vault-pass`. NO se une en el primer arranque
(sin vault disponible, riesgo de hostname no definitivo, join fallido no debe
arriesgar el despliegue). Alternativa más segura documentada:
`adcli preset-computer` + one-time password por equipo.

### Ficheros tocados

- **Nuevos**: `roles/preparaAD/{defaults/main.yml, tasks/main.yml,
  templates/krb5.conf.j2, templates/10-iac-ad.conf.j2, CLAUDE.md,
  LeemeComoUnirAlDominio.md}` (este último: guía operativa para humanos —
  qué hace el rol, cómo comprobar la unión y procedimiento recomendado).
- **`roles.yaml`**: `preparaAD` añadido activo (último, tras `vmware`);

- **Documentación**: tabla de roles y "Por hacer" de `Ubuntu/ansible/CLAUDE.md`;

## 18:29 — preparaAD: arreglo nsswitch.conf para dominios `.local` (mDNS cortaba ping/kinit/adcli)

**Motivo**: diagnóstico del usuario — el rol decía "Dominio visible por DNS:
SÍ" pero `ping iesmhp.local` no resolvía. Causa: hay **dos rutas de
resolución** distintas. `realm discover` y SSSD consultan systemd-resolved
directamente (ahí el split-DNS del paso 6 funciona), pero `ping`, `kinit` y
`adcli` usan `getaddrinfo()` → `/etc/nsswitch.conf`, cuya línea
`hosts: files mdns4_minimal [NOTFOUND=return] dns` intercepta CUALQUIER
nombre `*.local` (TLD reservado para mDNS, RFC 6762): el mDNS no encuentra
`iesmhp.local`, el `[NOTFOUND=return]` corta la cadena y `dns` (resolved)
nunca llega a consultarse. NO era solo cosmético: la unión usa kinit/adcli
por esa misma ruta y podía fallar.

### `roles/preparaAD/tasks/main.yml` (nuevo paso 6b)

- Tarea `replace` sobre `/etc/nsswitch.conf`: reordena la línea a
  `hosts: files dns mdns4_minimal [NOTFOUND=return]` — solo si
  `preparaad_arregla_nsswitch=true` (default) y el dominio termina en
  `.local`. Idempotente (tras el cambio el regex ya no casa); sin reinicios
  (glibc relee el orden en la siguiente resolución). Los nombres mDNS
  legítimos (impresoras…) siguen funcionando: resolved contesta "no existe"
  al instante a los `.local` sin routing domain y la cadena continúa hasta
  `mdns4_minimal`.
- Resumen: línea nueva "nsswitch hosts (.local/mDNS)" con tres estados —
  AJUSTADO en este pase / OK / AVISO si una línea `hosts:` no estándar dejó
  el mDNS por delante (el regex no casó; ajustar a mano).
- Nota: el reorden solo sirve junto con el drop-in split-DNS del paso 6:
  resolved se niega a enviar `.local` a DNS unicast salvo que exista la
  routing domain `~iesmhp.local` (que pone el drop-in). Son las dos mitades
  del mismo arreglo.
