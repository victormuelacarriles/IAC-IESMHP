# 2026-06-16 — Cambios

## Hostname por defecto: `mint` → `ld` + fecha de instalación

**Síntoma**: al instalar con la ISO personalizada, los equipos cuya MAC NO está
en `macs.csv` quedaban con el nombre `mint` (residuo de la versión Linux Mint,
ya anotado como pendiente en `20260611-Cambios.md` líneas 34-37).

**Causa**: dos sitios fijan el hostname en la cadena Ubuntu:
- `2-SetupSOdesdeLiveCD.sh` (chroot, durante la instalación) → defecto `$DISTRO`
  = `Ubuntu`.
- `Ubuntu/ISO/26.04/utiles/NombreIP.sh` (primer arranque, vía
  `3-SetupPrimerInicio.sh`) → defecto `mint`. Al ejecutarse DESPUÉS, este
  sobrescribía el `Ubuntu` del chroot, así que el nombre final visible era
  `mint`.

**Objetivo**: que esos equipos se llamen `ld` + `AAAAMMDDHHMM` (p. ej.
`ld202606160934` para una instalación del 16/06/2026 a las 9:34), con un nombre
**único por equipo** y **estable** (no debe cambiar en cada arranque).

**Fix**:
- `2-SetupSOdesdeLiveCD.sh` (paso "MAC, hostname y claves SSH autorizadas"):
  el defecto pasa de `EQUIPOENMACS=$DISTRO` a
  `EQUIPOENMACS="ld$(date +%Y%m%d%H%M)"`. El nombre se genera **una sola vez**
  aquí (momento de la instalación). Mensajes de error del bloque actualizados a
  `'$EQUIPOENMACS'`.
- `NombreIP.sh` (primer arranque): el defecto deja de ser `"mint"`. Ahora
  **conserva** el hostname asignado en la instalación
  (`EQUIPOENMACS="$(hostname)"`), de modo que el `ld+fecha` no se regenera en
  cada arranque. Auto-reparación: si encuentra un nombre genérico
  (`ubuntu`/`Ubuntu`/`mint`/`Mint`/`localhost`/vacío), genera uno nuevo
  `ld$(date +%Y%m%d%H%M)`.

**Comportamiento resultante**:
- MAC en `macs.csv` → hostname de `macs.csv` (sin cambios).
- MAC ausente → `ld<fecha de instalación>` fijado en el chroot y conservado en
  el primer arranque.
- `AULA` (derivada del nombre con `cut -d'-' -f1`) no coincide con `IABD`/`SMRD`
  para un nombre `ld...`, así que el equipo no registrado se queda en DHCP —
  mismo comportamiento que antes con `mint`.

**Nota de despliegue**: ambos ficheros viven en el repo que `0b-Github.sh` clona
desde GitHub al arrancar la ISO; requieren commit+push a `main` antes de generar
o arrancar una ISO nueva.

## `0b-Github.sh` — hora correcta (zona Europe/Madrid + NTP) antes de continuar

**Síntoma**: en el Live CD la hora aparecía 2 h adelantada.

**Causa**: típico desfase RTC/UTC en VMware con host Windows — el RTC lleva hora
local de Madrid, pero Linux lo interpreta como UTC y le suma el offset de la zona
(en verano CEST = UTC+2 → +2 h). Sin zona horaria fijada ni NTP, ese desfase se
arrastraba a los timestamps de todos los logs y ficheros creados por los scripts
posteriores.

**Fix**: nuevo bloque en `0b-Github.sh` justo **después** de confirmar
conectividad y **antes** del `git clone`/`apt` (así la hora correcta beneficia
también al clonado y a `apt`):
- `timedatectl set-timezone Europe/Madrid`.
- `timedatectl set-ntp true` (servidores NTP por defecto de `systemd-timesyncd`,
  vía `FallbackNTP`) + `systemctl restart systemd-timesyncd`.
- Bucle de espera (máx ~30 s, 15×2 s) sondeando
  `timedatectl show -p NTPSynchronized --value` hasta `yes`.

**Comportamiento ante fallo**: si NTP no confirma sincronización en el tiempo
máximo (p. ej. UDP 123 bloqueado por firewall/proxy de aula), **no aborta** la
instalación: registra un `[!]` con la hora actual y continúa. No se modifica el
RTC (`set-local-rtc`), porque la sincronización NTP corrige la hora del sistema
con independencia de cómo se interprete el RTC.

**Nota de despliegue**: `0b-Github.sh` se embebe en la ISO por `0a-CreaISO.sh`;
este cambio requiere **regenerar la ISO** (no se clona desde GitHub: `0b` ya está
dentro del squashfs).

**Resultado en VM (16/06)**: en `0b-Github.sh` el NTP **no** confirmó
sincronización dentro de los ~30 s (aparece el aviso `[!] NTP no confirmó...`),
aunque la zona sí quedó fijada (la hora se mostró como `... CEST`). Se decide
**no tocar `0b-Github.sh`** y repetir la misma comprobación más adelante.

## `1-SetupLiveCD.sh` — misma comprobación de hora al inicio

**Motivo**: en `0b-Github.sh` el NTP a veces no ha confirmado sincronización a
tiempo. Se **replica el mismo bloque** (zona Europe/Madrid + `set-ntp` + restart
`systemd-timesyncd` + bucle de espera ~30 s) al inicio de `1-SetupLiveCD.sh`,
tras crear las carpetas de trabajo y antes de detectar discos. Para ese punto la
red lleva más tiempo lista, así que la sincronización suele cuajar. Adaptado al
estilo del script (`echoverde`/`echoamarillo`, `set -e`). **No** se modifica
`0b-Github.sh`. Mismo comportamiento ante fallo: avisa y continúa, no aborta.

Ambos scripts (`0b` y `1`) van embebidos/clonados en la ISO; `1-SetupLiveCD.sh`
**sí** se clona desde GitHub (requiere commit+push a `main`), pero `0b` ya está
en el squashfs — para esta tanda basta regenerar la ISO o, si solo cambia `1`,
commit+push.

## `0b-Github.sh` y `1-SetupLiveCD.sh` — fallback HTTP para la hora (~13:10)

**Síntoma (VM, 16/06 12:54)**: igual que en `0b-Github.sh`, el bloque de hora de
`1-SetupLiveCD.sh` tampoco logra sincronizar por NTP (`[!] NTP no confirmó
sincronización (¿UDP 123 bloqueado?)`). La zona horaria sí queda bien fijada
(`Europe/Madrid`, hora mostrada como `... CEST`), pero la hora del sistema puede
seguir desfasada porque NTP no llegó a corregirla.

**Causa**: en las aulas el firewall/proxy filtra el **UDP 123** (NTP), así que
`systemd-timesyncd` nunca confirma `NTPSynchronized=yes`. El **TCP 443** (HTTPS)
sí está abierto — de hecho `0b-Github.sh` clona el repo y `apt` descarga por ahí.

**Fix**: tras el bucle de espera NTP, si **no** cuajó, segundo intento por HTTP en
la rama `else` de **ambos** scripts:
- Se hace `curl -sI` (o `wget -SqO /dev/null` como alternativa) a
  `https://www.google.com`, `https://github.com` y `https://www.cloudflare.com`
  (se prueban en orden hasta que uno responda).
- Se extrae la cabecera de respuesta `Date:` (viene siempre en **GMT**) y se fija
  la hora con `date -s`; como la zona ya es `Europe/Madrid`, `date` la convierte
  sola a la hora local. Luego `hwclock --systohc` propaga al RTC.
- Si tampoco hay red HTTP, se avisa y se continúa (no aborta, igual que antes).

**Detalles de implementación**:
- `1-SetupLiveCD.sh` usa `set -e` (sin `pipefail`): el pipeline de parseo solo
  depende del exit de `sed` (0). Estilo `echoverde`/`echoamarillo`.
- `0b-Github.sh` usa `set -euo pipefail`: cada pipeline de parseo lleva `|| true`
  para que un `grep` sin coincidencia o un `SIGPIPE` de `head` no aborte el
  script. Estilo `log`/`warn`.

**Nota de despliegue**: `1-SetupLiveCD.sh` se clona desde GitHub (requiere
commit+push a `main`); `0b-Github.sh` va embebido en el squashfs, así que su
cambio requiere **regenerar la ISO**.

## `3-SetupPrimerInicio.sh` + rol Ansible `horaHTTP` — hora correcta TRAS reinicios (~13:40)

**Síntoma**: el fallback HTTP funciona durante la instalación, pero **tras los
reinicios la hora vuelve a estar mal**.

**Causa**: dos cosas combinadas. (1) En las aulas el **UDP 123 (NTP)** está
filtrado → `systemd-timesyncd` nunca corrige. (2) En **VMware con host Windows**
el RTC se reinterpreta en cada arranque (RTC en hora local leído como UTC). Los
scripts `0b`/`1`/`3` solo corrigen la hora **durante la instalación/primer
arranque**, y `3-SetupPrimerInicio.sh` **se autodeshabilita** → no queda nada que
recorrija la hora en los arranques siguientes.

**Fix en dos partes**:

1. **`3-SetupPrimerInicio.sh`** — se sustituyen las 3 líneas del bloque de hora
   (`set-timezone` + `set-ntp` sin esperar ni verificar) por el bloque completo
   NTP + espera ~30 s + **fallback HTTP** (cabecera `Date:` de HTTPS), igual que
   en `0b`/`1`. Adaptado a `echoverde`/`echoamarillo` y a `set -e` (sin
   `pipefail`). Corrige también el typo «Ajustadando».

2. **Rol Ansible nuevo `Ubuntu/ansible/roles/horaHTTP`** (la corrección
   **persistente**): instala
   - `/usr/local/sbin/iac-sincroniza-hora.sh` (mismo algoritmo NTP→HTTP, log con
     `logger -t iac-hora`),
   - `iac-sincroniza-hora.service` (oneshot, `After=network-online.target`,
     `WantedBy=multi-user.target`) → se ejecuta en **cada arranque**,
   - `iac-sincroniza-hora.timer` → re-sincroniza cada `horahttp_timer_intervalo`
     (default `1h`) para la deriva en equipos con mucho uptime.

   `defaults/main.yml` expone `horahttp_zona`, `horahttp_urls`,
   `horahttp_ntp_intentos`, `horahttp_timer_intervalo` y rutas. El rol no depende
   de `community.general` (usa `ansible.builtin.command` + `ansible.builtin.systemd`).
   Añadido a `roles.yaml` como **activo, 2º** (justo tras `basicos`, para que la
   hora quede bien antes de los roles que tocan apt/certificados). Documentado en
   `roles/horaHTTP/CLAUDE.md` y en las tablas/listas de `Ubuntu/ansible/CLAUDE.md`,
   `Ubuntu/CLAUDE.md` y `CLAUDE.md` (raíz).

**Validado**: `bash -n` de `3-SetupPrimerInicio.sh` y del script de plantilla
renderizado; YAML de `roles.yaml` y del rol parsea sin errores.

**Nota de despliegue**: ambos cambios viven en el repo que se clona desde GitHub
(requieren commit+push a `main`). `3-SetupPrimerInicio.sh` y el rol se aplican en
el primer arranque; el rol deja el service+timer instalados para los siguientes.
Diagnóstico en el equipo: `journalctl -t iac-hora`,
`systemctl status iac-sincroniza-hora.service`,
`systemctl list-timers iac-sincroniza-hora.timer`.

## SSH de usuarios del dominio AD — `UsePAM yes` (~16:49)

**Síntoma (VM/PC `IABD-08`, 16/06)**: un usuario del dominio (`victor`) entra por
GDM y por RDP, pero **no por SSH**. `ssh victor@10.0.72.128` rechaza la
contraseña correcta. En `journalctl -u ssh`:
```
error: Could not get shadow information for victor
Failed password for victor from ... ssh2
```
`getent passwd victor` **sí** resuelve (uid 457001117), o sea que SSSD/nsswitch
van bien.

**Causa**: `sshd -T | grep usepam` → **`usepam no`**. El `/etc/ssh/sshd_config`
de Ubuntu 26.04 quedó reducido a `Include /etc/ssh/sshd_config.d/*.conf` +
`#PasswordAuthentication yes`, **sin** línea `UsePAM` → cae al **default
compilado de OpenSSH, que es `no`**. Con `UsePAM no`, sshd valida la contraseña
contra el `/etc/shadow` local; un usuario del dominio no está en shadow (su
contraseña vive en AD y se comprueba por PAM→`pam_sss`) → falla siempre. GDM y
RDP funcionan porque **sí** pasan por PAM. **No** era el formato del usuario:
con `use_fully_qualified_names=False` (snippet `10-iac-ad.conf`) se entra como
`victor` a secas, ni `MHPIES\victor` ni `victor@dominio`.

**Fix** (drop-in `/etc/ssh/sshd_config.d/60-iac-ad.conf` con `UsePAM yes` +
`PasswordAuthentication yes` + `KbdInteractiveAuthentication yes`, 0644, +
`restart ssh`), en **dos sitios con el MISMO fichero/contenido** (mantener en
sync):
- **`Ubuntu/ISO/26.04/3-SetupPrimerInicio.sh`** (primer arranque): tras el `sed`
  de `PermitRootLogin`, escribe el drop-in con un heredoc antes del
  `service ssh restart` existente.
- **Rol Ansible `preparaAD`** (`tasks/main.yml`, nuevo **paso 2b**, tras
  `pam_mkhomedir`): `ansible.builtin.copy` del drop-in + `systemd` `restarted`
  solo si cambió. Se aplica **siempre** (no solo con el equipo unido; `UsePAM
  yes` es el valor seguro por defecto y queda idempotente respecto al script).
  Las conexiones SSH activas sobreviven al restart (no corta un pase de aula).

**Documentación**: nuevo punto **2b** en "Prerequisitos que implanta" y nota en
"Estado / Notas" de `roles/preparaAD/CLAUDE.md`.

**Verificado**: `sshd -T | grep -i usepam` → `yes` y `ssh victor@10.0.72.128`
entra correctamente tras aplicar el drop-in a mano en `IABD-08`.

**Nota de despliegue**: el rol y `3-SetupPrimerInicio.sh` se clonan desde GitHub
(requieren commit+push a `main`). Diagnóstico: `sshd -T | grep -i usepam` (debe
ser `yes`); si falta `Include /etc/ssh/sshd_config.d/*.conf` en `sshd_config`,
el drop-in se ignora.

## Rol `Docker` — Docker rootless automático para CUALQUIER usuario, incl. dominio (~17:20)

**Síntoma (PC/VM `IABD-08`, 16/06)**: con un usuario nuevo (local) o de dominio
recién logueado, `docker run hello-world` falla:
```
failed to connect to the docker API at unix:///var/run/docker.sock; ...
dial unix /var/run/docker.sock: connect: no such file or directory
```

**Causa**: el daemon de **sistema** está desactivado a propósito (modo rootless,
rol `Docker`). Cada usuario necesita SU daemon rootless, pero eso lo hacían a
mano `rolesUsuario/DockerRootless` / `0-ConfiguracionInicial.sh` (PARTE 2), nunca
de forma automática. Un usuario que no los ejecutó no tiene daemon de usuario ni
contexto/`DOCKER_HOST` → el cliente cae al socket de sistema inexistente. Además,
`0-ConfiguracionInicial.sh` asigna subuid con `usermod --add-subuids`, que **solo
conoce `/etc/passwd` y FALLA con usuarios del dominio** (viven en SSSD).

**Fix** (todo en el rol `roles/Docker`, **nueva sección 6**, mecanismo elegido:
hook PAM para la parte root + servicio de usuario global para la parte usuario):
- **Parte ROOT — `templates/iac-docker-rootless-prep.sh.j2`** (a
  `/usr/local/sbin/`): lo invoca **`pam_exec`** en `common-session` en cada
  apertura de sesión (GDM/SSH/RDP/`su -`). Asigna subuid/subgid **escribiéndolos
  directamente a `/etc/subuid`+`/etc/subgid`** (rango único, `flock` para logins
  simultáneos — válido para locales y de dominio) y `loginctl enable-linger`.
  Idempotente, rápido en la ruta ya-hecha, salta root/sistema (`min_uid`), ignora
  `close_session` y siempre `exit 0`.
- **Activación PAM**: perfil pam-config `files/iac-docker-rootless`
  (`Session-Interactive-Only`, `optional pam_exec.so quiet …`) a
  `/usr/share/pam-configs/`, activado con `pam-auth-update --enable
  iac-docker-rootless` (solo si no está ya). Sobrevive a los `pam-auth-update`
  de `preparaAD` (mkhomedir/sss).
- **Parte USUARIO — `templates/iac-docker-rootless-user.sh.j2`** (a
  `/usr/local/bin/`): lo ejecuta el **servicio de usuario global**
  `files/iac-docker-rootless.service` (a `/etc/systemd/user/`, habilitado con
  `systemctl --global enable`). Espera ~15 s al subuid (race del 1er login),
  `dockerd-rootless-setuptool.sh install` (si falta la unit), `systemctl --user
  enable --now docker` y **`docker context use rootless`** (el cliente apunta al
  socket rootless **sin** `DOCKER_HOST` en `~/.bashrc`, así va en gnome-terminal
  y apps gráficas). Best-effort: reintenta en el siguiente login.
- **`defaults/main.yml`**: `docker_rootless_auto` (true), `docker_rootless_min_uid`
  (1000), `docker_subid_block_size` (65536), `docker_subid_block_start` (100000).
- **Seguridad**: NO se da sudo a los alumnos (la parte root la hace el hook PAM);
  `optional`+`quiet` ⇒ un fallo del helper nunca bloquea el login.
- **Primer arranque**: se usa `command` (`systemctl --global enable`,
  `pam-auth-update`), NO el módulo `systemd` con `enabled:`/`daemon_reload:` (que
  cuelga `3-SetupPrimerInicio`, bugs 2026-05-17). `--global enable` solo escribe
  symlinks en `/etc/systemd/user/`, sin contactar con ningún gestor.

**Coherencia del script manual `rolesUsuario/0-ConfiguracionInicial.sh`**: su
PARTE 2 (paso 2.2) usaba `sudo usermod --add-subuids 100000-165535`, que **falla
con usuarios del dominio** (usermod solo conoce `/etc/passwd`, no SSSD) y además
asignaba un rango FIJO (colisión si se aplica a varios usuarios). Ahora: asigna
subuid/subgid **a fichero** con rango único (sin solape) y **delega en el helper
del rol** `/usr/local/sbin/iac-docker-rootless-prep.sh` si está presente (subuid
+ lingering en un solo paso); si no, fallback inline vía `sudo tee`. Docstring de
cabecera y `rolesUsuario/CLAUDE.md` (paso 7) actualizados. `bash -n` → OK.

**Documentación**: sección nueva + tabla de variables + paso 6 + issue conocido en
`roles/Docker/CLAUDE.md`.

**Validado**: `bash -n` de los dos scripts renderizados (sustituyendo las vars
Jinja) y `yaml.safe_load` de `tasks/main.yml` + `defaults/main.yml` → OK. Falta
**probar en equipo** (login de un usuario de dominio y `docker run hello-world`
en la 2ª sesión).

**Diagnóstico en el equipo**: `journalctl -t iac-docker-rootless-prep -t
iac-docker-rootless-user`, `grep iac-docker-rootless-prep
/etc/pam.d/common-session`, `getent subuid <usuario>`, `systemctl --global
is-enabled iac-docker-rootless.service`.

**Nota de despliegue**: el rol se clona desde GitHub (requiere commit+push a
`main`). En equipos ya instalados, reaplicar: `sudo ansible-playbook -i
localhost, --connection=local roles.yaml --tags docker`. Quien YA tuviera sesión
abierta debe **reabrir sesión** una vez (el hook PAM corre al abrir sesión).
