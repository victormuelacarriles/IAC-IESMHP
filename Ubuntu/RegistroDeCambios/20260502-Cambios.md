# Registro de cambios — 2026-05-02

## Problema: kernel panic "VFS: Unable to mount root fs on unknown-block(0,0)" pese a 0 errores

### Síntoma
`4-Comprobaciones.sh` reportaba `0 error(es)` al final del chroot, pero al reiniciar
el equipo aparecía:
```
KERNEL PANIC! VFS: Unable to mount root fs on unknown-block(0,0)
```

### Análisis
El script tenía tres gaps que impedían detectar la causa real:

1. **La sección 2 no mostraba la cmdline real del kernel**: el grep
   `grep -E 'menuentry|^\s+linux '` llenaba los 10 resultados con líneas `menuentry`,
   dejando fuera las líneas `linux` de las entradas de Ubuntu. No se veía qué `root=`
   se estaba pasando al kernel.

2. **Faltaba verificar que GRUB estuviera instalado en la EFI**: existía `grub.cfg`
   pero no se comprobaba si `grubx64.efi` y los módulos GRUB (`/boot/grub/x86_64-efi/`)
   estaban presentes. Sin ellos el firmware UEFI no puede cargar GRUB.

3. **No se cruzaba el UUID de grub.cfg con el de fstab**: si `update-grub` escribía
   un UUID incorrecto, el script solo comprobaba que ese UUID existiera en algún
   dispositivo, pero no que coincidiera con el `/` de fstab.

### Solución aplicada

**Archivo:** `Ubuntu/ISO/26.04/4-Comprobaciones.sh` — versión `1.1-20260502`

#### Cambio 1 — Sección 2 (GRUB): mostrar y verificar cmdline real del kernel
```bash
# Extraer la línea linux de la primera entrada Ubuntu (no del submenu)
LINUX_LINE=$(awk '/menuentry .Ubuntu[^,]/{f=1} f && /^\s+linux\s/{print; exit}' /boot/grub/grub.cfg)
```
- Se muestra la cmdline completa como `[INF]`
- Se verifica que `root=` use UUID (no `/dev/`) → `[ERR]` si usa `/dev/`
- Se cruza el UUID de grub.cfg con el UUID raíz de `/etc/fstab` → `[ERR]` si son distintos

#### Cambio 2 — Nueva sección 7: GRUB EFI instalado
Verifica:
- Que `/boot/efi/EFI/` exista y contenga `grubx64.efi` o `shimx64.efi`
- Que `/boot/grub/x86_64-efi/` tenga más de 10 módulos

Si alguno de estos falta → `[ERR]` con el `Fix:` exacto de `grub-install`.

#### Cambio 3 — Auto-reboot al final
- Si `$ERRORES -eq 0`: cuenta atrás de 5 segundos y `reboot` automático
- Si hay errores: `read -n 1` (pausa hasta tecla) para diagnóstico
