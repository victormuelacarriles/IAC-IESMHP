# Registro de cambios — 2026-05-02

## Problema: kernel panic "VFS: Unable to mount root fs on unknown-block(0,0)" pese a 0 errores

### Síntoma
`4-Comprobaciones.sh` reportaba `0 error(es)` al final del chroot, pero al reiniciar
el equipo aparecía:
```
KERNEL PANIC! VFS: Unable to mount root fs on unknown-block(0,0)
```

### Análisis
El script tenía tres gaps que impedían detectar la causa real:

1. **La sección 2 no mostraba la cmdline real del kernel**: el grep
   `grep -E 'menuentry|^\s+linux '` llenaba los 10 resultados con líneas `menuentry`,
   dejando fuera las líneas `linux` de las entradas de Ubuntu. No se veía qué `root=`
   se estaba pasando al kernel.

2. **Faltaba verificar que GRUB estuviera instalado en la EFI**: existía `grub.cfg`
   pero no se comprobaba si `grubx64.efi` y los módulos GRUB (`/boot/grub/x86_64-efi/`)
   estaban presentes. Sin ellos el firmware UEFI no puede cargar GRUB.

3. **No se cruzaba el UUID de grub.cfg con el de fstab**: si `update-grub` escribía
   un UUID incorrecto, el script solo comprobaba que ese UUID existiera en algún
   dispositivo, pero no que coincidiera con el `/` de fstab.

### Solución aplicada

**Archivo:** `Ubuntu/ISO/26.04/4-Comprobaciones.sh` — versión `1.1-20260502`

#### Cambio 1 — Sección 2 (GRUB): mostrar y verificar cmdline real del kernel
```bash
# Extraer la línea linux de la primera entrada Ubuntu (no del submenu)
LINUX_LINE=$(awk '/menuentry .Ubuntu[^,]/{f=1} f && /^\s+linux\s/{print; exit}' /boot/grub/grub.cfg)
```
- Se muestra la cmdline completa como `[INF]`
- Se verifica que `root=` use UUID (no `/dev/`) → `[ERR]` si usa `/dev/`
- Se cruza el UUID de grub.cfg con el UUID raíz de `/etc/fstab` → `[ERR]` si son distintos

#### Cambio 2 — Nueva sección 7: GRUB EFI instalado
Verifica:
- Que `/boot/efi/EFI/` exista y contenga `grubx64.efi` o `shimx64.efi`
- Que `/boot/grub/x86_64-efi/` tenga más de 10 módulos

Si alguno de estos falta → `[ERR]` con el `Fix:` exacto de `grub-install`.

#### Cambio 3 — Auto-reboot al final
- Si `$ERRORES -eq 0`: cuenta atrás de 5 segundos y `reboot` automático
- Si hay errores: `read -n 1` (pausa hasta tecla) para diagnóstico

### Archivos modificados
- `Ubuntu/ISO/26.04/4-Comprobaciones.sh` — secciones 2 y 7 (nueva), final del script

## Problema 2: kernel panic persiste pese a 0 errores — causa real: línea `initrd` ausente en grub.cfg

### Síntoma
Tras los fixes del problema 1, `4-Comprobaciones.sh` seguía reportando `0 error(es)` pero
el kernel panic `VFS: Unable to mount root fs on unknown-block(0,0)` continuaba al reiniciar.

### Causa raíz
**Orden de operaciones en `2-SetupSOdesdeLiveCD.sh`**: `update-grub` (paso 8) se ejecutaba
**antes** de que `update-initramfs` (paso 10) creara el fichero `/boot/initrd.img-7.0.0-14-generic`.

Cuando `update-grub` corre, el fichero de initramfs no existe (solo hay symlinks colgantes
`initrd.img → initrd.img-7.0.0-14-generic`). El script `/etc/grub.d/10_linux` usa `test -e`
para comprobar si existe el initramfs — como el symlink está colgante, `test -e` devuelve
`false` → **grub.cfg se genera sin la línea `initrd`**.

Al reiniciar:
1. GRUB carga el kernel pero no carga initramfs (línea `initrd` ausente)
2. El kernel recibe `root=UUID=...` pero sin initramfs no puede resolver el UUID
3. → `VFS: Unable to mount root fs on unknown-block(0,0)`

Los checks de `4-Comprobaciones.sh` no detectaban esto porque:
- Solo verificaban la línea `linux` del grub.cfg (que tenía el UUID correcto)
- No verificaban la existencia ni validez de la línea `initrd`
- El check de NVMe usaba `grep -q nvme` que matchea directorios, no solo `.ko` reales

### Solución aplicada

#### `2-SetupSOdesdeLiveCD.sh` — vs 22.4-20260502
Añadido al final del paso 10 (tras verificar que el initramfs existe):
```bash
if ! grep -qE '^\s+initrd\s' "$GRUB_CFG" 2>/dev/null; then
    info "  grub.cfg sin línea initrd — re-ejecutando update-grub..."
    update-grub ...
    sed -i "s|root=${ROOT_DEV_STEP10}\b|root=UUID=${ROOT_UUID_STEP10}|g" "$GRUB_CFG"
fi
grep -E '^\s*(linux|initrd)\s' "$GRUB_CFG" | head -8  # diagnóstico visible en log
```
Esto garantiza que grub.cfg siempre tiene la línea `initrd` apuntando al initramfs recién creado.

#### `4-Comprobaciones.sh` — vs 1.2-20260502
1. **Nuevo check: línea `initrd` en grub.cfg** (sección 2): detecta su ausencia como `[ERR]`
   con mensaje claro sobre el VFS panic.
2. **Check NVMe mejorado**: busca `nvme.*\.ko` en lugar de solo `nvme` para distinguir
   entre módulo real y mero directorio.

### Archivos modificados
- `Ubuntu/ISO/26.04/2-SetupSOdesdeLiveCD.sh` — paso 10, tras verificación de initramfs
- `Ubuntu/ISO/26.04/4-Comprobaciones.sh` — sección 2 (check initrd) y sección 1 (check NVMe)

## Problema 3: autostart del Live CD se ejecuta también en el primer boot del sistema instalado

### Síntoma
Al arrancar el sistema instalado por primera vez, aparecía un terminal ejecutando `iac-iesmhp-run.sh`
(que llama a `0b-Github.sh`) como si fuera el Live CD.

### Causa raíz
`0a-CreaISO.sh` escribe `iac-iesmhp-setup.desktop` en `/etc/skel/.config/autostart/` del squashfs.
`1-SetupLiveCD.sh` copia el squashfs al sistema instalado vía rsync, incluyendo ese fichero.
Cuando `2-SetupSOdesdeLiveCD.sh` crea el usuario `usuario` con `useradd -m`, el skel se copia al
home del nuevo usuario → GNOME arranca el autostart en cada login.

### Solución aplicada

**`2-SetupSOdesdeLiveCD.sh` — vs 22.5-20260502**

Nuevo paso "Eliminar autostart del Live CD del sistema instalado" insertado **antes** del paso
de creación de usuarios (para que el skel esté limpio cuando `useradd` lo copie):
```bash
for _autostart_file in \
    /etc/skel/.config/autostart/iac-iesmhp-setup.desktop \
    /home/ubuntu/.config/autostart/iac-iesmhp-setup.desktop
do
    [ -f "$_autostart_file" ] && rm -f "$_autostart_file" && ok "Eliminado: $_autostart_file"
done
```

## Problema 4: logos del boot (Plymouth) muestran los de Ubuntu por defecto

### Síntoma
Al arrancar el sistema instalado, la pantalla de splash de Plymouth mostraba el logo estándar
de Ubuntu en lugar de los logos personalizados del IES Miguel Herrero.

### Causa raíz
Los ficheros de tema Plymouth (`bgrt-fallback.png`, `watermark.png`) en
`/usr/share/plymouth/themes/spinner/` y `/usr/share/plymouth/themes/bgrt/` son los que viene
con Ubuntu. Los personalizados estaban disponibles en el repo pero no se copiaban al sistema.
Además, `update-initramfs` embebe esos ficheros en el initramfs, así que la copia debe hacerse
ANTES de ese paso.

### Solución aplicada

**`2-SetupSOdesdeLiveCD.sh` — vs 22.5-20260502**

Nuevo paso "Plymouth: logos personalizados IES Miguel Herrero" insertado justo ANTES del paso de
`update-initramfs`. Copia desde `$RAIZDISTRO/imagenesIES/` a todos los directorios de tema que
existan:
- `bgrt-fallback.png` → `/usr/share/plymouth/themes/spinner/` y `/usr/share/plymouth/themes/bgrt/`
- `watermark.png` → `/usr/share/plymouth/themes/spinner/`

Los ficheros fuente son `Ubuntu/ISO/26.04/imagenesIES/bgrt-fallback.png` y `watermark.png`,
que llegan al chroot porque `1-SetupLiveCD.sh` copia todo el repo a `/opt/IAC-IESMHP`.

## Problema 5: sistema instalado auto-logea como 'ubuntu' → terminal "cuenta no disponible"

### Síntoma
Tras la instalación, en lugar de mostrar la pantalla de login de GDM, el sistema arrancaba
directamente a una sesión GNOME. Al intentar abrir un terminal, aparecía:
> "Actualmente esta cuenta no está disponible."
Además, el fondo de escritorio era el estándar de Ubuntu (no el del IES).

### Causa raíz
El Live CD de Ubuntu configura GDM con `AutomaticLoginEnable=True` y `AutomaticLogin=ubuntu`
en `/etc/gdm3/custom.conf`. El `rsync` de `1-SetupLiveCD.sh` copia esta configuración al sistema
instalado. Al arrancar, GDM auto-logea como `ubuntu`, cuya cuenta en el sistema instalado puede
estar bloqueada (PAM rechaza la apertura de shell → "cuenta no disponible"). Además, el dconf
personal de `ubuntu` sobreescribe el override de sistema del fondo personalizado del IES.

### Solución aplicada

**`2-SetupSOdesdeLiveCD.sh` — nuevo paso "Eliminar usuario ubuntu (Live CD) y deshabilitar auto-login GDM"**
insertado entre el paso de Usuarios y el de GRUB:

1. `userdel -r ubuntu` — elimina el usuario del Live CD y su home del sistema instalado.
2. Modifica `/etc/gdm3/custom.conf` con `sed` para desactivar `AutomaticLoginEnable`.
   Si el fichero no existe, lo crea con los valores por defecto (auto-login deshabilitado).

Con este fix, el sistema instalado muestra la pantalla de login de GDM y el usuario `usuario`
(con contraseña `usuario`) puede iniciar sesión. Al ser un usuario nuevo sin dconf personal,
hereda el override del sistema-db → se muestra el fondo del IES.

### Archivos modificados
- `Ubuntu/ISO/26.04/2-SetupSOdesdeLiveCD.sh` — nuevo paso entre Usuarios y GRUB

## Problema 6 (corregido en v7): Live CD Plymouth sigue mostrando logos Ubuntu por defecto

### Síntoma
Aunque `0a-CreaISO.sh` no daba error, al arrancar desde la ISO personalizada los logos de
Plymouth (boot screen) seguían siendo los estándar de Ubuntu, no los del IES.

### Historia del análisis (dos iteraciones)

**Iteración 1 (Problema 6 original):** Se intentó APPEND con CPIO sin comprimir:
```
cat initrd_path overlay.cpio > patched   # ← append sin comprimir
```
Resultado: `Initramfs unpacking failed: invalid magic at start of compressed archive`.  
El kernel ve el magic CPIO (070701) tras el stream comprimido y lo interpreta como stream
comprimido → error. Fix incorrecto: se cambió a PREPEND.

**Iteración 2 (este fix):** PREPEND también era incorrecto porque el kernel procesa los
segmentos en orden y **el último sobreescribe los anteriores**. Con PREPEND:
1. Kernel extrae nuestro CPIO → imágenes IES en tmpfs ✓  
2. Kernel descomprime el initramfs principal (que va DESPUÉS) → imágenes Ubuntu **sobreescriben** las nuestras ✗  

Plymouth mostraba Ubuntu porque el initramfs principal ganaba al ser el último segmento.

### Causa raíz definitiva
Ningún problema de magic ni de formato: el orden importa. El último segmento es el que
"gana" para ficheros duplicados. Había que añadir nuestro overlay como **último segmento** y
con un formato comprimido para que el kernel lo reconozca como segmento nuevo (no raw CPIO).

### Solución aplicada

**`0a-CreaISO.sh` — `customize_plymouth_initrd`**: APPEND con gzip (las imágenes del IES van al final → ganan):
```bash
# Comprimir el overlay con gzip
gzip -9 -c "${overlay_cpio}" > "${overlay_cpio_gz}"
# Concatenar: initrd original + overlay gzip (el último segmento sobreescribe duplicados)
cat "${initrd_path}" "${overlay_cpio_gz}" > "${patched_initrd}"
cp "${patched_initrd}" "${initrd_path}"
```
- `gzip magic (1f 8b)` → el kernel lo reconoce como nuevo segmento comprimido, no da "invalid magic"
- Al ir AL FINAL, nuestras imágenes Plymouth sobreescriben las del initramfs principal ✓

### Archivos modificados
- `Ubuntu/ISO/26.04/0a-CreaISO.sh` — función `customize_plymouth_initrd`

## Problema 7: Sistema instalado arranca sin pantalla de login + "cuenta no disponible"

### Síntoma
Aunque el fix del Problema 5 deshabilitó el auto-login GDM correctamente (confirmado en log),
el sistema instalado seguía arrancando directamente al escritorio sin mostrar el selector de
usuarios. Al abrir un terminal en esa sesión aparecía:
> "Actualmente esta cuenta no está disponible."

### Causa raíz (dos factores independientes)

**Factor A — gnome-initial-setup intercepta el primer arranque:**  
Ubuntu 26.04 incluye `gnome-initial-setup`. Si el fichero
`/var/lib/gdm3/.config/gnome-initial-setup-done` no existe, GDM lanza el asistente de
bienvenida en lugar de la pantalla de login. El asistente corre como un pseudo-usuario
(`gnome-initial-setup`) que no puede abrir un terminal de sistema → "cuenta no disponible".  
El fix del Problema 5 desactivó correctamente el `AutomaticLogin` pero no creó este flag.

**Factor B — `dconf update` fallaba en el chroot por `xkb-options=[]`:**  
En `00-language`, el valor `xkb-options=[]` es un array no tipado → `unable to infer type`.
`dconf update` abortaba con error, dejando sin compilar la base de datos GDM
(`/etc/dconf/db/gdm`). Con el perfil GDM apuntando a `system-db:gdm` y la base sin existir,
GDM podía comportarse de forma inesperada (degraded mode o fall-through a initial setup).

**Factor C — sed frágil para GDM:**  
El `sed -i` para desactivar `AutomaticLoginEnable` devuelve siempre exit code 0, incluso si
no hubo ninguna sustitución. Si el formato del fichero en el squashfs difería (tabulaciones,
mayúsculas distintas), el auto-login seguía activo sin que el log lo detectara.

### Solución aplicada

**`2-SetupSOdesdeLiveCD.sh` — vs 22.7-20260502**

1. **Fix Factor B** — eliminar `xkb-options=[]` del dconf language:
   El valor correcto para array vacío es `@as []`, pero lo más simple es omitir la clave
   (el default ya es vacío). Sin esta línea, `dconf update` compila correctamente.

2. **Fix Factor C** — sobreescribir `/etc/gdm3/custom.conf` en lugar de parchearlo con sed:
   ```bash
   cat > /etc/gdm3/custom.conf << 'EOF'
   [daemon]
   AutomaticLoginEnable=False
   TimedLoginEnable=False
   ...
   EOF
   ```
   Garantiza que el fichero siempre queda con auto-login deshabilitado,
   independientemente del formato original del squashfs.

3. **Fix Factor A** — crear el flag de gnome-initial-setup:
   ```bash
   mkdir -p /var/lib/gdm3/.config
   touch /var/lib/gdm3/.config/gnome-initial-setup-done
   # También en skel (para futuros usuarios) y en /home/usuario
   ```

### Archivos modificados
- `Ubuntu/ISO/26.04/2-SetupSOdesdeLiveCD.sh` — PASO 3 (quitar xkb-options) y PASO 9 (GDM + gnome-initial-setup)

## Problema 8: Plymouth en modo texto en Live CD + GDM Greeter auto-login (regresiones de v7)

### Síntomas
1. Live CD arranca en modo texto ("Ubuntu 26.04" + puntos en esquina superior izquierda).
2. Sistema instalado muestra sesión abierta automáticamente; al bloquearse muestra "GDM Greeter".

### Causa raíz — Plymouth texto (regresión del fix de Problema 6 v7)

El `gzip-append` añadido en v7 degrada Plymouth a modo texto. Plymouth durante el Live CD
usa el tema `spinner`/`bgrt` del initramfs principal. Al añadir un segmento gzip con nuestras
imágenes al final, algo en la inicialización de Plymouth falla (posiblemente las imágenes
que reemplazamos no tienen el formato exacto que el tema espera, o el tamaño/perfil no es
compatible con la variante de Plymouth del kernel 7.0.x del Live CD). Plymouth no puede
renderizar en modo gráfico y cae en modo texto.

**Conclusión**: La personalización del initrd del Live CD es frágil y no merece el riesgo.
El Plymouth del sistema INSTALADO sí funciona correctamente (Problema 4). El del Live CD
(activo solo 2-3 min durante la instalación) no justifica el esfuerzo.

**Fix**: comentada la llamada a `customize_plymouth_initrd` en `main()` de `0a-CreaISO.sh`.
El Live CD vuelve a mostrar el Plymouth estándar de Ubuntu (gráfico, sin errores).

### Causa raíz — GDM Greeter auto-login (regresión del fix de Problema 7)

Ubuntu 26.04 Live CD puede configurar el auto-login de GDM usando **ficheros drop-in** en
`/etc/gdm3/custom.conf.d/` además de `custom.conf`. Al sobreescribir solo `custom.conf`,
los drop-ins seguían activos y re-habilitaban el auto-login como el usuario `gdm` (que corre
el greeter de GDM). Al bloquearse esa sesión, la pantalla de bloqueo mostraba "GDM Greeter".

Adicionalmente, el GDM greeter en Ubuntu 26.04 usa GNOME Shell en sesión Wayland (usuario `gdm`).
Si el screensaver se activa en esa sesión, aparece la pantalla de bloqueo con "GDM Greeter" y
el usuario no puede desbloquearla (no conoce la contraseña de `gdm`).

### Solución aplicada (vs 22.8-20260502)

**`0a-CreaISO.sh`**: `customize_plymouth_initrd` comentada en `main()`.

**`2-SetupSOdesdeLiveCD.sh`** — PASO 9:
1. **Eliminar drop-ins**: `rm -rf /etc/gdm3/custom.conf.d` antes de sobreescribir `custom.conf`.
   Log añadido para ver qué contenía el directorio.
2. **Deshabilitar screensaver del GDM greeter**: añadido a `gdm.d/00-login-screen`:
   ```
   [org/gnome/desktop/session]
   idle-delay=uint32 0
   [org/gnome/desktop/screensaver]
   lock-enabled=false
   ```
3. **`dconf update` al final de PASO 9**: recompila todas las bases dconf incluyendo los
   settings del screensaver (el primer `dconf update` corre en PASO 3, antes de que se
   añadan estos settings).

### Archivos modificados
- `Ubuntu/ISO/26.04/0a-CreaISO.sh` — `main()`: comentada llamada `customize_plymouth_initrd`
- `Ubuntu/ISO/26.04/2-SetupSOdesdeLiveCD.sh` — PASO 9 (drop-ins, screensaver, dconf update)
