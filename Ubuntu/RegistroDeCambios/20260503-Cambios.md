# Registro de cambios — 2026-05-03

## Problema 10: Auto-login persiste tras todos los fixes del chroot (v22.9)

### Síntoma
El sistema instalado arranca directamente al escritorio sin mostrar pantalla de login de GDM,
aunque el chroot (v22.9) confirmaba `AutomaticLoginEnable=False` en `/etc/gdm3/custom.conf`.

### Análisis del log (2-SetupSOdesdeLiveCD.sh v22.9)

El log confirmaba que PASO 9 se ejecutó correctamente:
- `AccountsService/users/` estaba **vacío** (el squashfs no incluye ese archivo — se crea en runtime)
- `custom.conf` se sobrescribió con `AutomaticLoginEnable=False` ✓
- `gnome-initial-setup-done` creado ✓
- Screensaver GDM deshabilitado ✓

Sin embargo el auto-login persistía. Hipótesis identificadas:

1. **`dpkg --configure -a`** en `3-SetupPrimerInicio.sh` (línea 95): el postinst de gdm3
   puede ejecutarse y leer los valores debconf del Live CD para regenerar `custom.conf`
   con `AutomaticLoginEnable=true`. El squashfs tiene valores debconf del Live CD configurados
   para auto-login de `ubuntu`.

2. **`apt-get full-upgrade`** sin `--force-confold`: si gdm3 se actualiza, su postinst
   puede reemplazar `custom.conf` con la versión del paquete (que en Ubuntu 26.04 tiene
   auto-login habilitado para la instalación por defecto).

3. **squashfs con auto-login**: el `rsync` de `1-SetupLiveCD.sh` copia el `custom.conf`
   del squashfs con `AutomaticLoginEnable=True`. Aunque el chroot lo fija, si algo
   interfiere antes o durante, el auto-login puede estar activo.

### Fixes aplicados (v22.10-20260503)

#### `0a-CreaISO.sh` — GDM fix en squashfs
Añadido en `customize_squashfs()` antes del `mksquashfs`:
- Se sobreescribe `/etc/gdm3/custom.conf` en el squashfs con `AutomaticLoginEnable=false`
- Esto garantiza que el `rsync` ya copia una config sin auto-login

#### `2-SetupSOdesdeLiveCD.sh` — Diagnósticos + debconf + `false` minúsculas
- Cambiado `False` → `false` en el valor del GDM config (compatibilidad)
- Añadido diagnóstico completo: muestra todos los ficheros en `/etc/gdm3/` con contenido,
  ficheros PAM `gdm*`, y estado debconf de gdm3
- Añadido `debconf-set-selections` para pre-configurar gdm3 con auto-login deshabilitado
  antes de que el postinst pueda leerlo

#### `3-SetupPrimerInicio.sh` — DEBIAN_FRONTEND + --force-confold + re-apply post-upgrade
- Añadido `export DEBIAN_FRONTEND=noninteractive` antes de `dpkg --configure -a`
- Añadido `debconf-set-selections` para gdm3 antes de `dpkg --configure -a`
- Añadido `-o Dpkg::Options::="--force-confold"` a `dpkg --configure -a`, `apt-get install` y `apt-get full-upgrade`
- Añadido bloque post-upgrade que sobreescribe `/etc/gdm3/custom.conf` después del upgrade

### Diagnóstico adicional añadido
El nuevo log de PASO 9 mostrará:
- Contenido de cada fichero en `/etc/gdm3/`
- Contenido de ficheros PAM `gdm*`
- Estado debconf de gdm3

Esto permitirá identificar exactamente qué causa el auto-login si vuelve a persistir.

## Problema 11: Dos fallos tras v22.10 (2026-05-03 ~09:00)

### Síntoma 1: Live CD no hace auto-login
El Live CD arrancaba mostrando iconos personalizados (Plymouth correcto), pero se quedaba
en la pantalla de login de GDM ("Live session user") en lugar de entrar automáticamente.
Al pulsar sobre el usuario, el proceso de instalación arrancaba correctamente.

### Síntoma 2: Sistema instalado entra al GDM Greeter automáticamente
Tras la instalación, el primer arranque del sistema instalado no mostraba la pantalla de
selección de usuario. En su lugar GDM entraba directamente al "GDM Greeter" (el asistente
de bienvenida `gnome-initial-setup` que GDM lanza como sesión propia).

### Análisis

**Síntoma 1 — causa**:  
El fix v22.10 escribía `AutomaticLoginEnable=false` en `/etc/gdm3/custom.conf` del squashfs.  
Casper (el sistema de Live CD de Ubuntu) configura el auto-login del Live CD escribiendo en
esa misma ruta en tiempo de ejecución. Nuestro valor `false` estático ganaba a la escritura
dinámica de casper → pantalla de login.  
El squashfs es fuente tanto del Live CD como del rsync al disco; la capa correcta para
deshabilitar auto-login en el sistema instalado es el chroot, no el squashfs.

**Síntoma 2 — causa**:  
El log de `2-SetupSOdesdeLiveCD.sh` confirmaba `AutomaticLoginEnable=false` ✓, usuario
registrado en AccountsService ✓, y 0 errores en Comprobaciones. El problema era que
Ubuntu 26.04 incluye `gnome-initial-setup` integrado en GDM: si GDM no tiene la clave
`InitialSetupEnable=false` en `custom.conf`, lanza el asistente de bienvenida como sesión
GDM propia (lo que el usuario ve como "GDM Greeter" automático) antes de mostrar la
pantalla de login. Esa clave no estaba en ninguno de los configs que escribíamos.

Adicionalmente, gnome-initial-setup >= 46 (Ubuntu 26.04) busca el marcador de "setup done"
en `~/.local/share/gnome-initial-setup-done` además de `~/.config/gnome-initial-setup-done`.
Solo teníamos la ruta antigua.

### Fixes aplicados (v22.10 → sin cambio de versión, misma sesión)

#### `0a-CreaISO.sh`
- **Eliminado** el bloque que escribía `AutomaticLoginEnable=false` en el squashfs.
  Comentario explicativo reemplaza el código removido.

#### `2-SetupSOdesdeLiveCD.sh`
- **Añadido** `InitialSetupEnable=false` al `[daemon]` del GDM custom.conf.
- **Añadidas** rutas nuevas del marcador gnome-initial-setup:
  - `/var/lib/gdm3/.local/share/gnome-initial-setup-done`
  - `/etc/skel/.local/share/gnome-initial-setup-done`
  - `/home/usuario/.local/share/gnome-initial-setup-done`

#### `3-SetupPrimerInicio.sh`
- **Añadido** `InitialSetupEnable=false` al bloque de re-aplicación post-upgrade.

## Problema 12: GDM Greeter en primer arranque — auto-login persiste (v22.11)

### Síntoma
El sistema instalado arranca y aparece "GDM Greeter" automáticamente sin mostrar la
pantalla de login, impidiendo abrir un terminal.

### Análisis de causa raíz

`3-SetupPrimerInicio.service` tiene `After=graphical.target` → GDM ya arrancó y leyó
`/etc/gdm3/custom.conf` **antes** de que el servicio pueda corregirlo. En el primer boot,
si algo sobreescribió `custom.conf` (casper.service en userspace, postinst de gdm3 durante
`dpkg --configure -a`, etc.), GDM lo lee con `AutomaticLoginEnable=true` y entra al
greeter de 'ubuntu' que no existe → sesión GDM Greeter sin usuario real.

### Fix aplicado (v22.11-20260503)

#### `2-SetupSOdesdeLiveCD.sh` — Nuevo servicio early-boot + enmascaramiento casper

1. **`iac-gdm-noautologin.service`** (nuevo): se instala en `/etc/systemd/system/` habilitado
   con `WantedBy=display-manager.service`. Clave: `Before=display-manager.service` +
   `After=local-fs.target`. En cada arranque, escribe `/etc/gdm3/custom.conf` con
   `AutomaticLoginEnable=false` + `InitialSetupEnable=false` ANTES de que GDM lo lea.
   Es un oneshot sin dependencias de red ni D-Bus → no puede bloquearse.

2. **`casper.service` enmascarado** vía `ln -sf /dev/null /etc/systemd/system/casper.service`:
   impide que casper tenga un servicio de userspace que re-habilite el auto-login.

### Por qué este fix es definitivo

```
boot sequence:
  local-fs.target
      └── iac-gdm-noautologin.service   ← escribe custom.conf = false
              └── display-manager.service (GDM)  ← lee custom.conf = false ✓
                      └── graphical.target
                              └── 3-SetupPrimerInicio.service  (llega tarde, pero el
                                  ← full-upgrade ya no importa porque el servicio
                                     early-boot corrige en el SIGUIENTE arranque también)
```

## Problema 13: Login como 'usuario' falla tras el primer arranque (v22.12)

### Síntoma
Tras el reboot, GDM muestra la pantalla de login correctamente (anti-autologin funciona).
Al intentar entrar como 'usuario' con contraseña 'usuario', GDM rechaza el login.

### Causa raíz

Ubuntu 26.04 aplica `pam_pwquality` en `chpasswd` con mínimo de 8 caracteres.  
Las contraseñas 'root' (4 chars) y 'usuario' (7 chars) son **rechazadas** por PAM:

```
CONTRASEÑA INCORRECTA: La contraseña tiene menos de 8 caracteres
```

`chpasswd` sale con código ≠ 0 pero el script **no verificaba el exit code** y siempre
imprimía `[OK]`. Las contraseñas quedaban sin establecer (account bloqueada: `!` en shadow).

Evidencia en el log (v22.11):
```
CONTRASEÑA INCORRECTA: La contraseña tiene menos de 8 caracteres
  [OK ]  09:38:38 Contraseña root establecida        ← FALSO [OK]
CONTRASEÑA INCORRECTA: La contraseña tiene menos de 8 caracteres
  [OK ]  09:38:38 Usuario 'usuario' en grupo sudo    ← FALSO [OK]
```

### Fix aplicado — `2-SetupSOdesdeLiveCD.sh` v22.12 (2026-05-03)

Líneas 245 y 254: sustituido `chpasswd` (con PAM) por `chpasswd -e` con hash SHA-512
pregenerado via `openssl passwd -6`. `chpasswd -e` escribe directamente en `/etc/shadow`
sin pasar por PAM quality checks.

```bash
# Antes (falla silenciosamente):
echo "root:root" | chpasswd
echo "usuario:usuario" | chpasswd

# Después (bypasea pam_pwquality):
echo "root:$(openssl passwd -6 'root')" | chpasswd -e
echo "usuario:$(openssl passwd -6 'usuario')" | chpasswd -e
```
