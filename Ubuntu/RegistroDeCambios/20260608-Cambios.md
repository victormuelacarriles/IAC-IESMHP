## 09:30 — Fix: `NO_PUBKEY` / keyring NVIDIA vacío por run fallida previa

**Fichero**: `Ubuntu/ansible/roles/nvidia/tasks/main.yml`.

Síntoma: `apt update` falla con `NO_PUBKEY DDCAE044F796ECB0` /
`repository ... is not signed`, y la tarea de descarga de la clave sale `ok`
(saltada) en vez de `changed`.

Causa: la **primera run fallida** (sin `curl`) ejecutó igualmente
`gpg --dearmor`, que **creó el keyring
`/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg` VACÍO**. El
`creates:` de la tarea de descarga lo daba por válido y saltaba la
regeneración → keyring vacío → apt no puede verificar la firma del repo.

Fix:
- Se quita el `creates:` del keyring (no distingue "existe" de "es válido").
- Nueva tarea `gpg --show-keys <keyring>` (`failed_when: false`) que detecta
  si el keyring contiene una clave válida.
- La descarga ahora corre `when: nct_keyring_check.rc != 0` (solo si falta o
  está vacío/corrupto → self-healing en máquinas que ya tienen el keyring
  malo), con `set -o pipefail` (bash) para que el pipe falle si `curl` falla
  y `gpg --dearmor --yes` para sobreescribir el fichero inválido.

## 10:05 — Eliminar `[DEPRECATION WARNING]` del módulo `mount` (rol `clienteNAS`)

**Fichero**: `Ubuntu/ansible/roles/clienteNAS/tasks/main.yml` (+ `CLAUDE.md`
del rol).

La tarea *Montar cada export del NAS* del rol `clienteNAS` mostraba varios
`[DEPRECATION WARNING]` en el log del primer arranque:

- `Importing 'to_bytes'/'to_native' from 'ansible.module_utils._text' is deprecated`
- `Passing 'warnings' to 'exit_json'/'fail_json' is deprecated`

No los genera la tarea del repo, sino el **código interno del propio módulo
`ansible.posix.mount`** (la colección instalada usa imports antiguos de
`ansible.module_utils`).

Primera intención (descartada): `deprecation_warnings = False` en un
`ansible.cfg` global. Se descarta porque silenciaría los avisos de **toda**
la fase Ansible, ocultando futuros warnings legítimos de otros roles.

**Fix acotado al rol**: se deja de usar el módulo `mount` (la fuente de los
avisos) y se gestiona el montaje a mano, en dos tareas idempotentes:

1. `lineinfile` escribe/actualiza la línea de cada export en `/etc/fstab`
   (`regexp` por punto de montaje + `regex_escape`).
2. `command: mount <punto>` monta **solo** los puntos que `nas_mp_check`
   (tarea previa `mountpoint -q`) detectó como no montados
   (`when: item.rc != 0`); los ya montados se omiten.

Así desaparecen los avisos sin tocar la configuración global y el resto de
deprecaciones siguen visibles. El `ansible.cfg` creado a las 10:05 se borró.

Se crea la carpeta hermana de `roles/` para agrupar las configuraciones que

(Ansible, **como root**, ámbito sistema), los scripts de `rolesUsuario/` se

- **`CLAUDE.md`**: documenta el propósito de la carpeta, la diferencia

- **`0-ConfiguracionInicial.sh`**: primer script. Idempotente y aborta si se
  lanza como root. (1) asegura `~/.ssh` 0700; (2) genera `~/.ssh/id_ed25519`

  `ssh localhost "exit 0"`. Salidas: `0` OK, `1` error (root/keygen), `2`

## 11:00 — Fix `0-ConfiguracionInicial.sh`: `~/.ssh` propiedad de root

**Fichero**: `Ubuntu/ansible/rolesUsuario/0-ConfiguracionInicial.sh`
(+ `CLAUDE.md` de la carpeta).

Síntoma en un equipo ya provisionado (IABD-14): al lanzarlo como `usuario`,
`chmod: Operation not permitted` y `ssh-keygen ... Saving key ... Permission
denied`. Causa: `~/.ssh` y `authorized_keys` pertenecen a **root:root** (el
instalador escribió las claves de admin/profes como root sin `chown` al
usuario), así que el usuario no puede ni `chmod` ni escribir su clave en su
propio `~/.ssh`.

**Fix (self-heal de propiedad)**: antes de operar, el script comprueba con
`[ -O "$SSH_DIR" ]` / `[ -O "$AUTH_KEYS" ]` si el usuario es el propietario.
Si no lo es, reasigna la propiedad con `sudo chown -R "$USUARIO:$GRUPO"
"$SSH_DIR"` (el contenido se conserva → no se pierden las claves ya
autorizadas) y continúa el resto como usuario. Si falta `sudo`, imprime el
`chown` exacto a ejecutar y aborta. Además, la creación de `~/.ssh` ahora
reintenta con `sudo` si el home pertenece a root. Coherente con la convención
de la carpeta ("nunca como root, pero pedir privilegios puntualmente").

## 11:15 — Fix de ORIGEN: `~/.ssh` de 'usuario' creado como root en el chroot

**Fichero**: `Ubuntu/ISO/26.04/2-SetupSOdesdeLiveCD.sh`.

Causa raíz del fallo anterior (no solo el síntoma). El bloque que copia
`authorized_keys` al usuario hacía:

```sh
mkdir -p /home/usuario/.ssh
cp /root/.ssh/authorized_keys /home/usuario/.ssh/
chown usuario:usuario /home/usuario/.ssh/authorized_keys   # ← solo el FICHERO
chmod 600 /home/usuario/.ssh/authorized_keys
```

Como el script corre **como root en el chroot**, `mkdir` crea el directorio
`/home/usuario/.ssh` con propietario **root:root**, y el `chown` solo
reasignaba el fichero `authorized_keys`, no el directorio. Resultado: en el
equipo instalado el usuario no podía hacer `chmod` ni escribir su propia clave
en `~/.ssh` (operación no permitida / permiso denegado).

**Fix**: `chown -R usuario:usuario /home/usuario/.ssh` (directorio incluido) +
`chmod 700` al directorio, antes del `chmod 600` del fichero. Así los equipos
nuevos nacen ya con `~/.ssh` propiedad del usuario. (El self-heal de las 11:00
en `0-ConfiguracionInicial.sh` sigue siendo útil para los equipos **ya
desplegados** con el bug; este fix evita que vuelva a ocurrir en ISOs nuevas.)

## 12:10 — Nuevo playbook `rolesUsuario.yaml` + rol `DockerRootless`

**Ficheros** (nuevos): `Ubuntu/ansible/rolesUsuario/rolesUsuario.yaml`,
`Ubuntu/ansible/rolesUsuario/roles/DockerRootless/{tasks,defaults}/main.yml`,
`Ubuntu/ansible/rolesUsuario/roles/DockerRootless/CLAUDE.md`. **Modificado**:
`Ubuntu/ansible/rolesUsuario/CLAUDE.md` (sección de contenido + tabla de roles).

Primer playbook de la carpeta `rolesUsuario/` (configuración **por usuario**).
A diferencia de `../roles.yaml` (root sobre el equipo), `rolesUsuario.yaml`
usa `hosts: localhost`, `connection: local` y **`become: no` global**; solo
las tareas que tocan el sistema usan `become: yes` puntual (de ahí el `-K` al
invocar). Sus roles viven en `rolesUsuario/roles/`.

**Rol `DockerRootless`** — instala Docker en **modo rootless** para el usuario
que lanza el playbook, según la guía oficial
(<https://docs.docker.com/engine/security/rootless/>):

1. Aborta si uid 0 (es un rol de usuario, no de root).
2. Repo oficial de Docker (`become: yes`): keyring en
   `/etc/apt/keyrings/docker.asc` + entrada APT.
3. Instala `docker-ce`, `docker-ce-cli`, `containerd.io`,
   `docker-buildx-plugin`, `docker-compose-plugin`, `docker-ce-rootless-extras`
   + prerrequisitos `uidmap`, `dbus-user-session`, `slirp4netns`,
   `fuse-overlayfs`.
4. Desactiva el daemon de Docker de **sistema** (configurable).
5. Asigna subuid/subgid (`usermod --add-subuids/--add-subgids`) si faltan.
6. `loginctl enable-linger` (el daemon de usuario arranca en el boot).
7. `dockerd-rootless-setuptool.sh install` **como el usuario** (idempotente:
   se salta si ya existe `~/.config/systemd/user/docker.service`).
8. `~/.bashrc`: exporta `PATH` y `DOCKER_HOST=unix:///run/user/<uid>/docker.sock`.
9. `systemctl --user enable --now docker` + verificación `docker info`
   (`failed_when: false`).

**Variables** (`defaults/`): usuario/uid/home desde facts, `dr_codename`
(codename del repo, autodetectado), listas de paquetes, `dr_subid_range`,
`dr_disable_system_docker`.

**Issue documentado**: Ubuntu 26.04 `resolute` es muy reciente y Docker puede
no publicar aún ese codename → `apt update` fallaría. Workaround:
`-e dr_codename=noble` al invocar. Pendiente de verificar en el aula qué
codename sirve. El playbook **no está enganchado** al primer arranque todavía
(se lanza a mano mientras se prueba).
