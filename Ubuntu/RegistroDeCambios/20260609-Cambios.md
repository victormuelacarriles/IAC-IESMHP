## Helper `nuevo-alumno.sh`: usuarios sin `sudo` ni `docker` por defecto

**Fichero**: `Ubuntu/ISO/26.04/2-SetupSOdesdeLiveCD.sh` (heredoc del helper
`/usr/local/sbin/nuevo-alumno.sh`).

Antes, el helper creaba al alumno con `GROUPS_EXTRA="sudo"` y añadía
dinámicamente `vboxusers/libvirt/docker` si existían (`useradd -G "$GROUPS_EXTRA"`).
Decisión del IES: los alumnos **no** deben tener privilegios de administrador ni
acceso a Docker por defecto.

- Eliminado el bloque `GROUPS_EXTRA` y el bucle de grupos opcionales.
- `useradd` pasa a ejecutarse **sin `-G`**: el usuario queda únicamente en su
  grupo primario propio (= nombre de usuario), creado por `useradd`.
- `echo` final del helper: `grupos: $GROUPS_EXTRA` → `solo grupo propio '$U'`.

Para conceder privilegios puntuales a posteriori:
`sudo usermod -aG sudo <usuario>` (admin) / `sudo usermod -aG docker <usuario>`.

**Equipo ya instalado con la versión antigua del helper** (p. ej. IABD-14):
corregir el usuario afectado sin reinstalar:
```bash
zfs set quota=none rpool/home/<u>   # quitar la cuota de 200G heredada
usermod -G <u> <u>                  # dejarlo solo en su grupo propio
```

## `0-ConfiguracionInicial.sh`: PARTE 2 — Docker rootless por usuario (≈ 10:10)

**Fichero**: `Ubuntu/ansible/rolesUsuario/0-ConfiguracionInicial.sh`
(+ `Ubuntu/ansible/rolesUsuario/CLAUDE.md`).

**Síntoma** (usuario nuevo `openclaw` en IABD-14):
```
$ docker compose up -d
unable to get image '...': failed to connect to the docker API at
unix:///var/run/docker.sock; ... dial unix /var/run/docker.sock:
connect: no such file or directory
```
El cliente apunta al socket del **daemon de SISTEMA** (`/var/run/docker.sock`),
que en este despliegue está **desactivado a propósito** (modo rootless). El
usuario nuevo nunca completó la parte rootless **por usuario**: no tiene
`DOCKER_HOST` en `~/.bashrc`, ni la `docker.service` de usuario, ni
(posiblemente) subuid/subgid ni lingering. En el usuario que sí funciona, eso ya
se hizo en su día.

**Fix**: el script tenía solo la PARTE 1 (SSH a sí mismo, que terminaba con
`exit 0/2`). Se reestructura para guardar el resultado SSH en `SSH_OK` (sin
salir) y se añade una **PARTE 2** que replica en bash el rol
[`DockerRootless`](../ansible/rolesUsuario/roles/DockerRootless/CLAUDE.md), pero
autosuficiente:

1. Comprueba que existe `dockerd-rootless-setuptool.sh` (parte de sistema del rol
   `roles/Docker`); si falta, avisa y sale `2`.
2. subuid/subgid: si faltan en `/etc/subuid`+`/etc/subgid`, los asigna con
   `sudo usermod --add-subuids/--add-subgids 100000-165535`.
3. Lingering: `sudo loginctl enable-linger <usuario>` y espera (≤10 s) a que
   exista `XDG_RUNTIME_DIR` (`/run/user/<uid>`).
4. `dockerd-rootless-setuptool.sh install` (idempotente: se salta si ya existe
   `~/.config/systemd/user/docker.service`).
5. Añade a `~/.bashrc` un bloque marcado `# >>> IAC-IESMHP DockerRootless >>>`
   con `export PATH=/usr/bin:$PATH` y
   `export DOCKER_HOST=unix:///run/user/<uid>/docker.sock` (mismo contenido que
   el `blockinfile` del rol, no se duplica).
6. `systemctl --user enable --now docker.service` + verificación `docker info`.

**Códigos de salida nuevos**: `0` = SSH verificado **y** `docker info` OK · `2` =
parte(s) sin verificar (sshd no disponible, o falta la parte de sistema de
Docker) · `3` = fallo en PARTE 2 (subuid/subgid o `dockerd-rootless-setuptool.sh`).

Tras ejecutarlo hay que **abrir un terminal nuevo** para que `~/.bashrc` exporte
`DOCKER_HOST`; luego `docker run --rm hello-world`.

`bash -n` sobre el script: sintaxis OK.
