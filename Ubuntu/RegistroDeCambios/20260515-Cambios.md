## 2-SetupSOdesdeLiveCD.sh — v22.17-20260515

### Hora: ~10:30

**Motivación**: tras instalar el SO y arrancar por primera vez, el usuario `usuario` puede iniciar sesión correctamente, pero al abrir un terminal y ejecutar `sudo apt update` se obtiene el siguiente error:

```
Err:2 file:/cdrom resolute Release
  Fichero no encontrado - /cdrom/dists/resolute/Release (2: No existe el archivo o el directorio)
Error: El repositorio «file:/cdrom resolute Release» no tiene un fichero de Publicación.
Información: No se puede actualizar de un repositorio como este de forma segura y por tanto está deshabilitado por omisión.
```

### Diagnóstico

El Live CD de Ubuntu añade automáticamente una entrada APT que apunta al ISO montado en `/cdrom` (entrada `deb cdrom:[Ubuntu ...]/ resolute main`). Esa entrada vive en `/etc/apt/sources.list` (formato clásico) o en `/etc/apt/sources.list.d/*.sources` (DEB822, Ubuntu 26.04).

El rsync de `1-SetupLiveCD.sh` copia esa configuración tal cual al sistema instalado. Tras el reboot, el sistema instalado ya no tiene `/cdrom` montado, así que cualquier `apt update` falla con el error visto. El sistema sigue funcionando porque las otras fuentes (archive.ubuntu.com, security.ubuntu.com) sí responden, pero el aviso es ruidoso y entorpece a los usuarios.

### Cambios aplicados

1. **Nuevo paso "Limpiar fuentes APT del Live CD (cdrom:)"** en `2-SetupSOdesdeLiveCD.sh`, insertado tras la generación de `/etc/fstab` y antes de "Verificar conectividad a Internet". Se ejecuta dentro del chroot, sobre las fuentes APT del sistema **instalado**, no del Live CD.

2. **Dos capas de limpieza**:
   - **Formato clásico** (`/etc/apt/sources.list` y `/etc/apt/sources.list.d/*.list`): se eliminan con `sed` las líneas que coincidan con la regex `^[[:space:]]*deb(-src)?[[:space:]]+(\[[^]]*\][[:space:]]+)?cdrom:` (cubre `deb cdrom:`, `deb-src cdrom:` y variantes con opciones `[arch=...]`).
   - **Formato DEB822** (`/etc/apt/sources.list.d/*.sources`, Ubuntu 26.04): si el fichero contiene una línea `URIs:` con `cdrom:`, se elimina el fichero completo (un `.sources` puede tener varios stanzas, pero el cdrom típicamente está aislado en su propio fichero como `ubuntu-cdrom.sources`).

3. **Diagnóstico**: antes de borrar, se loguean las líneas/ficheros que se van a eliminar para dejar rastro en `2-SetupSOdesdeLiveCD.sh.log`. Al final del paso se reporta el número total de fuentes limpiadas; si es 0, se loguea con `[INF]` (no es un error: simplemente esa versión del Live CD no añadió cdrom).

4. **Versión** — `VERSIONSCRIPT` de `22.16-20260514` → `22.17-20260515`.

### Ficheros modificados

- `Ubuntu/ISO/26.04/2-SetupSOdesdeLiveCD.sh`
- `Ubuntu/CLAUDE.md` (sección "Bugs corregidos")
- `Ubuntu/RegistroDeCambios/20260515-Cambios.md` (nuevo)

### Cómo verificar tras regenerar la ISO

Dentro del sistema instalado, tras arrancar como `usuario`:

```bash

grep -rn 'cdrom:' /etc/apt/sources.list /etc/apt/sources.list.d/ 2>/dev/null

# 'apt update' debe completar sin el error "file:/cdrom":
sudo apt update
```

En el log de chroot (`/var/log/IAC-IESMHP/Ubuntu/2-SetupSOdesdeLiveCD.sh.log`) debe verse el paso "Limpiar fuentes APT del Live CD" con al menos un `[OK]` indicando el fichero/línea limpiada.

## 2-SetupSOdesdeLiveCD.sh — v22.18-20260515 (segunda iteración del mismo bug)

### Hora: tarde (segunda ronda de pruebas)

**Síntoma**: tras regenerar la ISO con la v22.17 e instalar el SO, `sudo apt update` sigue fallando con `Err:2 file:/cdrom resolute Release`. Sin embargo `grep -rn 'cdrom:' /etc/apt/sources.list /etc/apt/sources.list.d/` no devuelve nada en el sistema instalado.

### Diagnóstico

La regex de v22.17 buscaba el prefijo `cdrom:` (URI tradicional de `apt-cdrom`), pero el Live CD de Ubuntu 26.04 puede escribir el origen como `file:/cdrom` (URI tipo `file://`). En `.sources` DEB822 esto aparece como:

```
Types: deb
URIs: file:/cdrom
Suites: resolute
Components: main restricted
```

Ese formato **no casa** con `^URIs:[[:space:]]*cdrom:`, así que la limpieza de v22.17 ignoraba el fichero y la entrada llegaba intacta al sistema instalado.

Verificado en VMware: `sudo rm /etc/apt/sources.list.d/*cdrom*` resuelve el problema en caliente y `apt update` deja de fallar; el grep posterior por `cdrom:` (con dos puntos) no encuentra nada, confirmando que la entrada vive con la URI `file:/cdrom`.

### Cambios aplicados

1. **Regex ampliada** — variable común `_CDROM_URI='(cdrom:|file:/+cdrom)'` aplicada a las dos capas:
   - **Clásico** (`.list`): `^[[:space:]]*deb(-src)?[[:space:]]+(\[[^]]*\][[:space:]]+)?${_CDROM_URI}`
   - **DEB822** (`.sources`): `^URIs:[[:space:]]*${_CDROM_URI}` o `^URIs:.*[[:space:]]${_CDROM_URI}`

2. **Fallback defensivo por nombre de fichero** — para `.sources`, si el nombre contiene `cdrom` (p.ej. `ubuntu-cdrom.sources`) el fichero se borra aunque la URI no case con la regex. Ubuntu 26.04 aísla las fuentes del Live CD en un `.sources` propio, así que el nombre es una pista fiable.

3. **Versión** — `VERSIONSCRIPT` de `22.17-20260515` → `22.18-20260515`.

### Ficheros modificados

- `Ubuntu/ISO/26.04/2-SetupSOdesdeLiveCD.sh` (paso "Limpiar fuentes APT del Live CD")
- `Ubuntu/RegistroDeCambios/20260515-Cambios.md` (este bloque)

### Cómo verificar tras regenerar la ISO

```bash
# Ambos greps deben devolver vacío (cdrom: y file:/cdrom)
sudo grep -rn 'cdrom' /etc/apt/sources.list /etc/apt/sources.list.d/ 2>/dev/null

# Y apt update debe completar sin Err:2:
sudo apt update
```

En `2-SetupSOdesdeLiveCD.sh.log` el paso debe reportar `[OK] Eliminado: /etc/apt/sources.list.d/ubuntu-cdrom.sources` (o similar).

## 3-SetupPrimerInicio.sh — v22.19-20260515 (log duplicado + Ansible python interpreter)

### Hora: ~12:30

El sistema instala y arranca correctamente hasta `3-SetupPrimerInicio`. Dos problemas detectados en `3-SetupPrimerInicio.sh.log`:

### Problema 1 — Cada línea del log se graba DOS veces

**Diagnóstico**: doble sumidero hacia el mismo fichero.

1. El script hace `exec > >(tee -a "$FLOG") 2>&1` (escribe en `$FLOG` y reenvía a su stdout).
2. La unidad systemd `3-SetupPrimerInicio.service` (creada por `2-SetupSOdesdeLiveCD.sh`) tenía además `StandardOutput=append:$RAIZLOG/$SCRIPT3.log` y `StandardError=append:...`.

Resultado: cada línea la escribe el `tee` del script (copia 1) y, como el stdout del `tee` lo captura systemd y lo **vuelve a anexar al mismo fichero**, se graba otra vez (copia 2). Prueba definitiva en el log: la línea 1 (`echo "$SCRIPT3 (vs...)"`, que corre **antes** del `exec`) aparece **una sola vez**; todo lo posterior al `exec`, duplicado.

Foco secundario: la sección de `4-Comprobaciones.sh` se invocaba con `bash "$SCRIPT4" 2>&1 | tee -a "$FLOG"`, re-anexando a `$FLOG` cuando el stdout ya pasa por el `tee` del `exec` → esa sección se duplicaba aparte.

**Cambios aplicados**:
- `2-SetupSOdesdeLiveCD.sh`: en la unidad `3-SetupPrimerInicio.service`, `StandardOutput`/`StandardError` pasan de `append:$RAIZLOG/$SCRIPT3.log` a `journal`. El único escritor del fichero es ahora el `tee` del propio script (sigue yendo copia al journal vía systemd, sin duplicar el fichero). Comentario de aviso añadido para no reintroducir el bug.
- `3-SetupPrimerInicio.sh`: `bash "$SCRIPT4" 2>&1 | tee -a "$FLOG" || true` → `bash "$SCRIPT4" || true` (el `exec` inicial ya redirige a `$FLOG`).

### Problema 2 — Ansible falla: `The module interpreter '/usr/bin/python3.12' was not found`

**Diagnóstico**: `3-SetupPrimerInicio.sh` pasaba `-e 'ansible_python_interpreter=/usr/bin/python3.12'` codificado a fuego. Ubuntu 26.04 «resolute» ya no incluye `python3.12` (`/bin/sh: 1: /usr/bin/python3.12: not found`, rc=127) → `ansible-playbook` aborta con `failed=1, ok=0` en el primer `setup`. El mismo valor estaba codificado en las inventarios `Ubuntu/ansible/*.ini`.

**Cambios aplicados**:
- `3-SetupPrimerInicio.sh`: el interprete se resuelve en runtime — `PYINT="$(command -v python3 || echo /usr/bin/python3)"` y se pasa `-e "ansible_python_interpreter=$PYINT"`. El symlink `/usr/bin/python3` siempre apunta al Python por defecto de la distro, sea cual sea su versión. Se loguea el intérprete elegido con `echoverde`.
- `Ubuntu/ansible/equiposIABD.ini`, `equiposSMRD.ini`, `EquiposSMRD-alumnos2526.ini`: `ansible_python_interpreter=/usr/bin/python3.12` → `auto_silent` (Ansible auto-descubre el intérprete de cada host sin warning de deprecación). Las inventarios de `Mint/` y `Mint/ControlSMRD/` **no** se tocan (otra distro, fuera de alcance).
