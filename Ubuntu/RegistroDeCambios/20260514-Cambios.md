# Cambios 2026-05-14

## 2-SetupSOdesdeLiveCD.sh — v22.16-20260514

### Hora: ~17:00

**Motivación**: tras instalar el SO y arrancar por primera vez, la pantalla no muestra el formulario de login. En su lugar se ve un escritorio GNOME funcional ejecutándose como el usuario `gdm-greeter` (uid 60578). El usuario no puede iniciar sesión porque no hay ventana de login y desconoce la contraseña del usuario `gdm-greeter`.

### Diagnóstico

Ubuntu 26.04 introdujo un usuario de sistema `gdm-greeter` separado del usuario `gdm`. GDM lanza el greeter para este usuario como una sesión Wayland mediante `gdm-wayland-session gnome-session` (sin pasar `--session=`).

Cuando `gnome-session` se invoca sin `--session=`, lee el valor de la clave dconf `/org/gnome/desktop/session/session-name`. Si no está fijada, cae al valor por defecto del sistema, que en Ubuntu 26.04 es `'ubuntu'`. Resultado: el greeter carga `/usr/share/gnome-session/sessions/ubuntu.session` (escritorio Ubuntu normal con `gnome-shell --mode=ubuntu`) en lugar de `/usr/share/gnome-session/sessions/gnome-login.session` (el formulario de login, `Kiosk=true`, `gnome-shell --mode=gdm`).

**Evidencia recogida en VM**:
```
gdm-gre+ 2221 /usr/libexec/gdm-wayland-session gnome-session
gdm-gre+ 2297 /usr/libexec/gnome-session-service --session=ubuntu
gdm-gre+ 2305 /usr/bin/gnome-shell --mode=ubuntu
```

Y `loginctl list-sessions` mostraba sesión `c1` clase `greeter` corriendo bajo `gdm-greeter` en tty1 sin login prompt.

El script anterior creaba `/etc/dconf/db/gdm.d/00-login-screen` con logo, idle-delay y screensaver pero **no fijaba** `session-name`. Además creaba `/etc/dconf/profile/gdm` pero no `/etc/dconf/profile/gdm-greeter` — el nuevo usuario podía estar usando el profile por defecto sin ver el system-db `gdm`.

### Cambios aplicados

1. **AccountsService entry para `gdm-greeter`** — nuevo bloque después del registro de `usuario`:
   ```
   /var/lib/AccountsService/users/gdm-greeter
   [User]
   Session=gnome-login
   XSession=gnome-login
   SystemAccount=true
   ```
   Solo se crea si el usuario `gdm-greeter` existe (`id gdm-greeter` succeed). `SystemAccount=true` lo oculta del selector de usuarios.

2. **dconf system-db `gdm` ampliado** — se añade `session-name='gnome-login'` a la sección `[org/gnome/desktop/session]` en `/etc/dconf/db/gdm.d/00-login-screen`.

3. **Lock dconf** — nuevo `/etc/dconf/db/gdm.d/locks/00-session-name` con la ruta `/org/gnome/desktop/session/session-name` para que ningún user-db pueda sobreescribir el valor.

4. **Perfil dconf para `gdm-greeter`** — nuevo `/etc/dconf/profile/gdm-greeter`:
   ```
   user-db:user
   system-db:gdm
   ```
   Garantiza que el nuevo usuario del greeter lea el mismo system-db `gdm` que ya configurábamos para el usuario `gdm`.

5. **Versión** — `VERSIONSCRIPT` de `22.15-20260504` → `22.16-20260514`.

### Cómo verificar tras regenerar la ISO

Dentro del sistema instalado, tras arrancar:

```
# Debe mostrar 'gnome-login' (NO 'ubuntu'):
sudo -u gdm-greeter dconf read /org/gnome/desktop/session/session-name

# Debe listar la clave (bloqueada):
cat /etc/dconf/db/gdm.d/locks/00-session-name

# El proceso gnome-session debe ejecutarse con --session=gnome-login:
ps -ef | grep 'gnome-session-service --session'

# Y gnome-shell con --mode=gdm (no --mode=ubuntu):
ps -ef | grep 'gnome-shell --mode'
```
